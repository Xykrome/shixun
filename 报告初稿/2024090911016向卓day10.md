# Epoll 并发服务器 训练报告

## 一、今天完成了什么

今天完成了 W2D5 的两个实训内容：**实训内容2——使用 epoll 实现 TCP 聊天服务器 EpollServer**，以及 **Webserver V1.0——基于 epoll 的 HTTP 服务器**。

### 实训内容2：EpollServer

新建项目目录 `epoll_chat/`，包含 `include/epoll_server.h`（客户端结构体与常量定义，MAX_CLIENTS=64）、`src/epoll_server.c`（epoll 服务器核心）、`src/main.c`（入口）、`src/chat_client.c`（复用 W2D4 的客户端）、`Makefile` 和 `tests/test_day10.sh`。

**核心架构**：`socket → bind → listen → epoll_create1(0) → epoll_ctl(ADD listen_fd) → while(1) { epoll_wait() → 遍历就绪事件：如果 ready_fd == listen_fd 则 accept() 加入客户端列表并 epoll_ctl(ADD)；否则 recv() → 按 \n 分帧提取完整消息 → handle_client_message() → send() 回显确认 }`。

**与 W2D4 select 版本的关键对比**：

| 维度 | select (W2D4) | epoll (W2D5) |
|------|--------------|--------------|
| 创建实例 | `FD_ZERO(&master_set)` | `epoll_create1(0)` → 返回 epfd |
| 注册 fd | `FD_SET(fd, &master_set)` | `epoll_ctl(epfd, EPOLL_CTL_ADD, fd, &ev)` |
| 等待事件 | `select(max_fd+1, &read_fds, NULL, NULL, NULL)` | `epoll_wait(epfd, events, MAX_EVENTS, -1)` |
| 检测就绪 | 遍历所有 fd，`FD_ISSET(fd, &read_fds)` | 遍历 events 数组（只含就绪 fd） |
| 移除 fd | `FD_CLR(fd, &master_set)` | `epoll_ctl(epfd, EPOLL_CTL_DEL, fd, NULL)` |
| 每次重建集合 | 需要（select 会修改传入的 fd_set） | 不需要（epoll 在内核维护事件表） |
| 时间复杂度 | O(n)，n 为最大 fd 值 | O(1)，只处理就绪事件 |

epoll 的核心优势在于内核维护感兴趣的事件列表，不需要每次循环重建；`epoll_wait` 直接返回就绪事件数组，不需要遍历全部 fd 逐个 `FD_ISSET` 检查。消息分帧逻辑（`extract_messages`）与客户端管理（`add_client`/`remove_client`）结构与 select 版本一致，但 `add_client` 内部增加了 `epoll_ctl(ADD)` 注册，`remove_client` 内部改为 `epoll_ctl(DEL)` + `close`。

**事件结构体**：`struct epoll_event ev; ev.events = EPOLLIN; ev.data.fd = conn_fd;`。使用 LT（电平触发）模式（epoll 默认行为），未处理完的数据下次 `epoll_wait` 仍会通知，编程模型更安全。

对比验收标准的 5 项检查点：编译与启动（make clean && make 无错误，服务端显示 socket/bind/listen/epoll_create1 各步）、3 个客户端并发连接（连接数/IP/端口/fd 显示）、\n 消息分帧（粘包 "msg1\nmsg2\nmsg3\n" → 一次 recv 拆成 3 条独立 MSG，半包测试 "hel" + "lo\n" → 拼成 "hello"）、连接退出与清理（quit → epoll_ctl(DEL) + close + Active connections 递减，recv==0 → 同上）、epoll 代码检查（epoll_create1/epoll_ctl/epoll_wait 正确使用，LT+EPOLLIN，未使用 select/多线程/多进程），全部通过，自动测试脚本 16/16 项通过。

### Webserver V1.0

在既有 `mini_webserver/` 项目中新增 `include/epoll_server.h` 和 `src/epoll_server.c`，修改 `src/main.c`（新增 `serve-epoll <max_requests>` 命令分支）和 `Makefile`（加入 epoll_server.c 编译目标与 `make test9` 测试目标），新建 `tests/test_day10.sh`。

**核心架构**：`socket → bind → listen → epoll_create1(0) → epoll_ctl(ADD listen_fd) → while (request_count < max_requests) { epoll_wait() → 如果 listen_fd 就绪则 accept() → 将 client_fd 加入 epoll；否则 recv() HTTP 请求 → handle_request_string() 路由分发 → send() HTTP 响应 → epoll_ctl(DEL) + close(client_fd) → request_count++ } → 达到 max_requests 后正常退出`。

**路由规则**（复用既有的 `handle_request_string()`）：`/hello` → HTTP/1.1 200 OK + "Hello, Web!"；`/users/<name>` → 200 OK + FOUND/NOT_FOUND；其他路径 → 404 Not Found。

**技术限制**（验收标准明确要求）：纯 epoll + 单线程事件循环，未使用 select、多线程、多进程或线程池。每个连接处理一个 HTTP 请求后立即关闭（HTTP/1.0 语义），处理完 `max_requests` 个请求后服务器正常退出。

对比验收标准的 10 项检查点：编译与启动（`./mini_web_server serve-epoll 8`）、/hello 路由（curl -i → 200 OK + Hello, Web!）、用户路由（/users/zhangsan → 200 OK + FOUND zhangsan 123 111）、404 处理（/missing → 404 Not Found）、多请求处理（连续 3 个请求全部返回 200 OK）、请求数量控制（8 个请求后正常退出）、连接清理（每次处理完 epoll_ctl DEL + close）、epoll 代码检查（epoll_create1/epoll_ctl/epoll_wait/LT+EPOLLIN）、技术限制（无 select/多线程/多进程/线程池），全部通过，自动测试脚本 18/18 项通过。

## 二、遇到的一个问题、原因和解决方案

**问题**：Webserver V1.0 编译时，`tcp_pool_server.c`（V0.8 的既有文件）报错 `expected expression before '==' token`，导致 `make` 失败。

**原因**：定位到 `tcp_pool_server.c` 第 169 行，注释 `===== 步骤 7: 关闭线程池 =====` 前面缺少 `/*`。该行只有 `* =====` 而非 `/* =====`，是一个"悬挂的注释结尾"——编译器将 `*` 解析为乘法运算符，后续的 `==` 自然产生语法错误。该 bug 在 V0.8 开发时已存在，但因 V0.8 编译时未触发（可能是 Makefile 增量编译跳过了该文件的重新编译）而未被发现。

**解决方案**：在第 169 行开头补上缺失的 `/*`，改为 `/* ===== 步骤 7: 关闭线程池 ===== */`。修复后所有源文件编译通过。

## 三、还存在什么不足，下一步如何改进

1. **Webserver V1.0 不支持 HTTP Keep-Alive**：当前每个连接处理一个请求后立即关闭，浏览器访问时每个资源（HTML/CSS/JS/图片）都需要重新建立 TCP 连接。改进：在响应头中添加 `Connection: keep-alive`，处理完一个请求后不立即关闭，而是继续 `epoll_wait` 等待同一 fd 上的后续请求，实现 HTTP/1.1 持久连接。

2. **Webserver V1.0 仅支持 GET 方法**：`handle_request_string()` 对非 GET 请求返回 405 Method Not Allowed，不支持 POST 等常用方法。改进：增加 POST 方法解析，读取请求体（Content-Length），支持简单的表单提交。

3. **EpollServer 无广播功能**：与 select 版本相同，消息仅回显给发送方，其他客户端看不到。改进：收到消息后遍历所有客户端 `send()`，实现群聊；支持 `/msg <name> <text>` 私聊。

4. **未使用 ET 模式和 EPOLLONESHOT**：当前两个项目均使用默认的 LT（电平触发）模式。验收标准提到"ET 模式和 EPOLLONESHOT 作为扩展"。改进：添加命令行参数 `--et` 启用边缘触发模式（`EPOLLET`），配合非阻塞 I/O 和循环 recv/send 直到 EAGAIN，减少 epoll_wait 唤醒次数；对 Webserver 增加 `EPOLLONESHOT` 确保同一时刻只有一个线程处理一个 fd（为后续引入线程池做准备）。

5. **Webserver V1.0 不支持静态文件服务**：当前仅处理两个硬编码路由，无法像真正的 Web 服务器那样从 `www/` 目录读取 HTML/CSS/JS 文件返回给浏览器。改进：增加默认路由处理器，将 URL path 映射到文件系统路径（如 `www/`），根据文件扩展名设置正确的 `Content-Type`（`text/html`、`text/css`、`image/png` 等）。

6. **两个项目共用同一套客户端管理逻辑但代码重复**：`epoll_chat/src/epoll_server.c` 和 `mini_webserver/src/epoll_server.c` 中的 `add_client`、`remove_client`、`find_client_by_fd` 等函数高度相似。改进：抽取公共的 epoll 工具层（如 `epoll_utils.h/c`），统一管理客户端数组、epoll 事件注册和连接清理，上层只需传入不同的事件处理回调函数。
