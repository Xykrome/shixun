# Webserver V1.3 训练报告

## 一、今天完成了什么

今天完成了 W3D3 的实训内容：**构建 HTTP 动态查询服务器 V1.3 —— 在 V1.2 静态文件服务的基础上增加 GET 和 POST 两种方式的学生信息查询，支持三种查询模式（仅班级、仅关键词、班级+关键词），并实现了与指导书一致的 UESTC 风格前端页面**。

W3D2 解决了"怎样安全地把请求转换为静态文件响应"，W3D3 解决的是"怎样处理动态请求——解析查询参数、校验输入、查询数据源并生成动态 HTML，同时让同一查询支持 GET 和 POST 两种提交方式"。核心新增模块 `query_handler.h/c`（约 650 行）负责 URL 解码、查询字符串解析、HTML 转义、参数校验、数据文件查询、跨班级搜索和动态页面生成；`http_server.c` 的路由分发增加 `/search` 路径识别、POST 请求体校验和 403/413/415 错误处理。

**新增文件**：`include/query_handler.h`、`src/query_handler.c`（动态查询处理器），`data/2011.txt`、`data/2012.txt`（制表符分隔的学生数据，每行格式为 `学号\t姓名\t性别`），`tests/test_day13.sh`（40 项自动化测试）。

**修改文件**：`src/http_server.c`（路由分发重构——GET `/search` 识别查询参数、POST `/search` 检查 Content-Type（→415）、Content-Length（→400/413）、请求体上限 4096 字节；405 响应 Allow 头动态适配 `/search` 路径为 `GET, POST`；版本号更新到 V1.3），`src/query_handler.c`（响应发送由 `snprintf` 到小缓冲区改为 `send_response()` 头体分离发送，解决截断问题；HTML 页面从蓝色居中表单重写为 UESTC 风格——白色 header、大标题、GET/POST 双按钮、表单结果同一页），`www/index.html`、`www/css/style.css`（静态首页同步更新为 V1.3 UESTC 风格，header 与查询页面保持一致），`include/http_server.h`（更新注释），`Makefile`（SRCS 增加 `query_handler.c`，新增 `test12` 目标），`src/main.c`（更新命令行帮助为 V1.3）。

**与 V1.2 的关键对比**：

| 维度 | V1.2 (W3D2) | V1.3 (W3D3) |
|------|------------|------------|
| 路由类型 | 仅静态文件 + POST /echo | 静态文件 + 动态查询 + POST /echo |
| GET 请求 | 全部 → `serve_static_file()` | `/search` → 动态查询；其他 → 静态文件 |
| POST 请求 | 仅 `/echo` | `/search`（表单查询）+ `/echo`（兼容） |
| 参数来源 | 无 | GET：URL 查询字符串；POST：`application/x-www-form-urlencoded` 请求体 |
| 查询模式 | 无 | 三种：仅班级（全班）/ 仅关键词（跨班级）/ 班级+关键词 |
| URL 处理 | 仅路径映射 | URL 解码（%XX → 字节，+ → 空格） |
| 输入校验 | 无 | class：4 位数字；keyword：1-64 字节，拒绝控制字符/`..`/`/`/`\`/空字节 |
| 数据源 | 无 | `data/<class>.txt` 制表符分隔文件 + 跨班级搜索 `query_all_classes()` |
| HTML 生成 | 由文件系统提供 | 动态生成：UESTC 风格页面（表单+结果合并，GET/POST 双按钮，JavaScript 提交） |
| XSS 防护 | 无 | `html_escape()` 转义 `&` `<` `>` `"` `'` |
| 状态码 | 200/400/403/404/405/500 | 新增 413（请求体过大）、415（不支持的 Content-Type）；403 同时覆盖静态文件和 data 目录 |
| Allow 头 | 固定 `Allow: GET` | `/search` 路径返回 `Allow: GET, POST` |
| 前端设计 | 紫色渐变卡片 | UESTC 风格：白色 header、1100px 宽、`#2f6db5` 蓝点缀、纯黑表头、`#f8f9fa` 底色 |

**动态查询处理器**（`query_handler.h/c`）实现了九个核心函数：

- `url_decode(src, dst, dst_size)`：将 `%XX` 编码转换为对应字节，`+` 转换为空格。支持 UTF-8 多字节序列（如 `%E7%94%B7` → 0xE7 0x94 0xB7 = "男"），非法 `%` 序列返回 -1。

- `parse_query_string(query, class_buf, keyword_buf)`：解析 `class=2011&keyword=%E7%94%B7` 格式的查询字符串。遍历 `&` 分隔的参数对，提取 key 和 value，对 value 执行 URL 解码后写入对应缓冲区。class 和 keyword 均为可选参数，至少需提供一个。

- `html_escape(src, dst, dst_size)`：将 `&` `<` `>` `"` `'` 分别替换为 `&amp;` `&lt;` `&gt;` `&quot;` `&#39;`。所有输出到 HTML 页面的用户数据（班级编号、关键字、学号、姓名、性别）均经过此函数转义，防止 XSS 注入。

- `validate_class(class_str)`：class 必须恰好 4 位数字（0-9），如 `2011`。不符合返回 400。

- `validate_keyword(keyword)`：长度 1-64 字节。拒绝空字节、控制字符（0x01-0x1F，TAB 除外）、路径穿越字符（`/`、`\`）和 `..`。UTF-8 多字节字符（0x80-0xFF）允许通过。

- `query_records(class_str, keyword, results_buf, match_count)`：构建路径 `data/<class>.txt` → `realpath()` 安全校验 → `fgets()` 逐行读取 → `strstr()` 匹配 keyword。返回值：0 成功、-1 文件不存在 → 404、-2 路径越界 → 403。keyword 为空时匹配所有行（class-only 模式）。匹配行按制表符拆分为学号/姓名/性别，经 `html_escape()` 转义后生成 `<tr>` 行。

- `query_all_classes(keyword, results_buf, match_count)`：仅关键词模式专用——遍历 `data/` 目录下所有 `.txt` 文件，在每个文件中搜索 keyword，每行结果附加班级列，生成四列表格行（班级/学号/姓名/性别）。同样对每个文件执行 `realpath()` 安全校验。

- `send_response(client_fd, status_code, reason, mime, body, body_len)`：内部辅助函数——先 `snprintf` 生成短响应头（状态行 + Content-Type + Content-Length + Connection，约 200 字节），`send_all()` 发送；再 `send_all()` 单独发送 HTML 响应体。彻底避免了 V1.2 中响应头与体合并到大缓冲区导致截断的问题。

- `handle_search_request(client_fd, method, query_str, ...)`：`/search` 路由主入口。四种分支：(1) GET 无参数 → 搜索表单页；(2) 仅 class → 调用 `query_records(keyword="")` 显示全班；(3) 仅 keyword → 调用 `query_all_classes()` 跨班级搜索；(4) class + keyword → `query_records()` 精确查询。所有分支通过 `send_response()` 统一发送，`access_log()` + `log_info()` 统一记录。

**三种查询模式的设计思路**：指导书要求 class 和 keyword 均为查询参数，但实际使用场景中用户可能只知道班级（想浏览该班所有学生）或只知道姓名（想查某个学生但不确定班级）。因此将两个参数均设为可选——至少提供一个即可查询。仅班级时显示该班全部学生，仅关键词时扫描所有班级文件并附加班级列方便区分，两者都提供时精确匹配。

**UESTC 风格前端页面**：根据指导书 PDF 中截图的设计规范（通过 ChatGPT 分析图片得到），搜索页面采用电子科技大学风格——白色 header 含 logo + "DYNAMIC SEARCH" + 导航（静态首页/动态查询）；主区域 900px 宽，`W3D3 · GET AND POST` 蓝色小字 + "学生信息查询" 72px 大标题 + "mini_webserver V1.3" 灰色版本号；表单和查询结果在同一页——班级输入框 + 关键词输入框 + **GET 查询**（蓝底白字 `#2f6db5`）+ **POST 查询**（白底黑框）两个按钮；结果区 `RESULT` 蓝色标签 + 纯黑表头表格（`#000` 背景白色文字）。两个按钮通过 JavaScript 实现——GET 按钮 `window.location.href` 跳转带参数 URL，POST 按钮动态创建 `<form method="POST">` 并 `submit()`。查询后输入框自动预填充上次的值，结果直接显示在表单下方。

**POST 请求体安全处理**：在 `http_server.c` 中 POST `/search` 路由先遍历请求头查找 `Content-Type`——不存在或非 `application/x-www-form-urlencoded` 返回 415；`Content-Length` 如果大于 `MAX_BODY_SIZE`（4096）返回 413；`Content-Length` 缺失或为 0 返回 400。通过 V1.1 已有的 `is_request_complete()` + `parse_http_request()` 机制确保请求体按 Content-Length 完整接收后才进入业务处理，防止半包数据被当作完整请求。

对比指导书 8 项具体实现要求的全部检查点：

1. **保留 V1.2 的 epoll 事件循环、静态资源处理、日志和统一 fd 清理**：`http_server.c` 中 epoll_create1/epoll_ctl/epoll_wait 30 处、`serve_static_file()` 调用 4 处、access_log/log_info 17 处、EPOLL_CTL_DEL+close 2 处。V1.2 静态文件测试 58/58 全部通过。

2. **GET /search 返回查询表单；参数统一使用 class 和 keyword**：`GET /search` → 返回 UESTC 风格表单。参数名严格使用 `class` 和 `keyword`（`parse_query_string()` 只识别这两个 key）。

3. **POST /search 解析 application/x-www-form-urlencoded 请求体**：POST 路由在 `http_server.c:450-545`，检查 Content-Type → 解析请求体 → 传入 `handle_search_request()`。GET 和 POST 查询同一份数据，返回相同结果。

4. **根据 \r\n\r\n 和 Content-Length 累计完整请求；请求体上限 4096 字节**：`http_parser.c` 中 `find_header_end()` + `get_content_length()` + `is_request_complete()` 确保请求完整。`MAX_BODY_SIZE=4096` 定义在 `query_handler.h`，POST 超限返回 413。

5. **URL 解码、UTF-8 和参数格式校验；班级只映射到固定 data 目录**：`url_decode()` 处理 `%XX` 和 `+`；validate 函数校验后 class 拼接为 `data/<class>.txt`，`realpath()` 验证路径仍在 `data/` 内。

6. **查询 data/<class>.txt 并生成 HTML 结果页；中文正常显示，输出内容须转义**：`query_records()` 读取制表符分隔文件；`html_escape()` 对所有字段转义后生成 `<tr>` 行；结果页通过 `generate_result_page_html()` 生成完整 HTML，Content-Type 显式 `charset=utf-8`。

7. **正确设置 Content-Type、Content-Length、Connection，使用 send_all() 发送**：`send_response()` 生成 `Content-Type: text/html; charset=utf-8` + `Content-Length: <body_bytes>` + `Connection: close`，头体分开发送均调用 `send_all()`。

8. **正确返回 400、403、404、405、413、415、500，保留日志和统一清理**：8 种状态码全部覆盖——400（参数格式/班级/keyword/缺 Content-Length）、403（静态文件 `realpath` 越界 + data 目录 `realpath` 越界 → `query_records` 返回 -2）、404（路由不存在 + 数据文件不存在）、405（非 GET/POST 访问 /search，Allow 头动态适配）、413（请求体 > 4096）、415（Content-Type 不正确）、500（内部 I/O 错误）。每分支统一执行 access_log + log_info + epoll_ctl(DEL) + close(client_fd)。

全部通过，自动测试脚本 **V1.3 40/40、V1.2 58/58、V1.1 38/38，合计 136 项全部通过**。

## 二、遇到的一个问题、原因和解决方案

**问题**：GET /search 和 GET /search?class=2011&keyword=男 返回的 HTML 页面中，表单元素和查询结果表格完全丢失，但错误页面（400/404）正常显示。

**原因**：`handle_search_request()` 最初的设计将 HTTP 响应头和 HTML 响应体一起放入 `char header[1024]` 缓冲区通过 `snprintf` 拼接后发送。搜索表单 HTML（约 1800 字节）和查询结果页 HTML（约 2000 字节）均远超 1024 字节，导致 `snprintf` 截断。更严重的是，`snprintf` 返回值是"如果缓冲区足够大本应写入的字节数"而非实际写入字节数，`send_all(client_fd, header, header_len)` 试图发送超过实际缓冲区大小的数据。客户端只收到响应头 + CSS 样式的前半部分（约 1000 字节处被截断），`<form>`、`<table>` 等核心内容完全丢失。错误页面 HTML 较短（约 500 字节），恰好未被截断，因此只有错误页面正常。

**解决方案**：新增内部辅助函数 `send_response(int client_fd, int status_code, const char *reason, const char *mime, const char *body, int body_len)`，将响应头发送和响应体发送彻底分离——先用 `snprintf` 将状态行 + Content-Type + Content-Length + Connection 格式化为短头部（`char header[512]`，不含响应体，永远够用），`send_all()` 发送；再 `send_all()` 单独发送 HTML 响应体。两个 `send_all` 的总发送量即为完整 HTTP 响应。`handle_search_request()` 的六个分支（表单、参数错误、class 错误、keyword 错误、404、200 结果）全部改为先 `generate_*_html()` 生成完整 HTML → `strlen()` 精确获取字节数 → `send_response()` 分两段发送。这个重构同时消除了六处重复的 `snprintf` + `send_all` 模式，所有错误处理和正常流程共享同一个发送逻辑。

## 三、还存在什么不足，下一步如何改进

1. **仅支持精确子串匹配，不支持模糊查询**：当前 `strstr()` 只能做大小写敏感的字节匹配。用户输入"张"能匹配"张三"，但输入拼音"zhang"、缩写"zs"或同音字均无法匹配。改进：增加拼音首字母索引（如"张三"→"zs"），支持通配符 `*` 和 `?`，对 keyword 同时尝试原始匹配和拼音匹配，取并集返回。

2. **数据文件每次请求重新打开读取**：`query_records()` 和 `query_all_classes()` 每次查询都 `fopen()` + 逐行 `fgets()` + `fclose()`。对于只有 8 条记录的文件影响不大，但数据量增长后磁盘 I/O 成为瓶颈。改进：服务器启动时加载所有 `data/*.txt` 到内存链表结构（按班级索引），查询时直接遍历内存；启动 `inotify` 监听 data 目录变化自动热重载。

3. **跨班级搜索的班级列表硬编码**：`query_all_classes()` 中班级列表为 `{"2011", "2012", NULL}` 硬编码，新增班级文件（如 `data/2013.txt`）不会被搜索到。改进：使用 `opendir()`/`readdir()` 动态遍历 `data/` 目录，提取所有 `.txt` 文件名作为班级列表，自动适配新增班级数据。

4. **无分页机制**：仅班级查询（如 `class=2011`）返回全部 8 条记录尚可接受，但跨班级搜索（如 `keyword=男` 返回 9 条）或数据量增大后页面会很长。改进：增加 `page` 和 `page_size` 参数，每页 20 条，生成"上一页/下一页"导航，只查询当前页所需的数据行。

5. **POST 按钮通过 JavaScript 动态创建 form 提交，依赖浏览器 JS 支持**：当前 POST 按钮的 `onclick` 使用 `document.createElement('form')` + `submit()`，如果浏览器禁用 JavaScript 则 POST 按钮完全无效。改进：在页面中同时放置一个隐藏的 `<form method="POST">`，POST 按钮的 onclick 先尝试填充该表单值再 submit，如果没有 JS 则 GET 按钮仍可通过普通链接工作。

6. **中文查询结果的 URL 编码体验不佳**：GET 查询提交后，浏览器地址栏显示 `%E7%94%B7` 而非"男"。虽然功能正确，但用户不直观。改进：生成结果页时在页面中增加一个"分享此查询"链接，显示解码后的查询条件文字说明；同时保留原始编码 URL 用于复制分享。
