# Webserver V1.3 训练报告

## 一、今天完成了什么

今天完成了 W3D3 的实训内容：**构建 HTTP 动态查询服务器 V1.3 —— 在 V1.2 静态文件服务的基础上增加 GET 和 POST 两种方式的学生信息查询功能**。

W3D2 解决了"怎样安全地把请求转换为静态文件响应"，W3D3 解决的是"怎样处理动态请求——解析查询参数、校验输入、查询数据源并生成动态 HTML"。核心新增模块 `query_handler.h/c`（约 450 行）负责 URL 解码、查询字符串解析、HTML 转义、参数校验、数据文件查询和动态页面生成；`http_server.c` 的路由分发增加 `/search` 路径识别和 POST 请求体校验（Content-Type、Content-Length、413/415 错误处理）。

**新增文件**：`include/query_handler.h`、`src/query_handler.c`（动态查询处理器），`data/2011.txt`、`data/2012.txt`（制表符分隔的学生数据），`tests/test_day13.sh`（38 项自动化测试）。

**修改文件**：`src/http_server.c`（路由分发重构——GET/POST `/search` 路由识别、POST Content-Type 和 Content-Length 校验、413/415 错误响应、405 响应 Allow 头动态适配 `/search` 路径、版本号更新到 V1.3），`include/http_server.h`（更新注释），`Makefile`（SRCS 增加 `query_handler.c`，新增 `test12` 目标），`src/main.c`（更新命令行帮助为 V1.3）。

**与 V1.2 的关键对比**：

| 维度 | V1.2 (W3D2) | V1.3 (W3D3) |
|------|------------|------------|
| 路由类型 | 仅静态文件 + POST /echo | 静态文件 + 动态查询 + POST /echo |
| GET 请求 | 全部 → `serve_static_file()` | `/search` → 动态查询；其他 → 静态文件 |
| POST 请求 | 仅 `/echo` | `/search`（表单查询）+ `/echo`（兼容） |
| 参数来源 | 无 | GET：URL 查询字符串；POST：请求体 |
| URL 处理 | 仅路径映射 | URL 解码（%XX → 字节，+ → 空格） |
| 输入校验 | 无 | class：4 位数字；keyword：1-64 字节，拒绝控制字符/../\\/空字节 |
| 数据源 | 无 | `data/<class>.txt` 制表符分隔文件 |
| HTML 生成 | 无（由文件系统提供） | 动态生成：搜索表单 / 查询结果表 / 错误页 |
| XSS 防护 | 无 | `html_escape()` 转义 & < > " ' |
| 状态码 | 200/400/403/404/405/500 | 新增 413（请求体过大）、415（不支持的 Content-Type） |
| Allow 头 | 固定 `Allow: GET` | `/search` 路径返回 `Allow: GET, POST` |

**动态查询处理器**（`query_handler.h/c`）实现了七个核心函数：

- `url_decode(src, dst, dst_size)`：将 `%XX` 编码转换为对应字节，`+` 转换为空格。支持 UTF-8 多字节序列（如 `%E7%94%B7` → 0xE7 0x94 0xB7 = "男"），非法 `%` 序列返回 -1。

- `parse_query_string(query, class_buf, keyword_buf)`：解析 `class=2011&keyword=%E7%94%B7` 格式的查询字符串。遍历 `&` 分隔的参数对，提取 key 和 value，对 value 执行 URL 解码，按参数名匹配后写入对应缓冲区。必须同时提取到 class 和 keyword 才算成功。

- `html_escape(src, dst, dst_size)`：将 `&` `<` `>` `"` `'` 分别替换为 `&amp;` `&lt;` `&gt;` `&quot;` `&#39;`。防止用户输入中的特殊字符破坏 HTML 结构或注入恶意脚本。输出到 HTML 页面的所有用户数据（班级编号、关键字、学号、姓名、性别）均经过此函数转义。

- `validate_class(class_str)`：class 必须恰好 4 位数字（0-9），如 `2011`。不符合的（如 `abc`、`20`）返回 400 错误。

- `validate_keyword(keyword)`：keyword 长度 1-64 字节。拒绝空字节、控制字符（0x01-0x1F，TAB 除外）、路径穿越字符（`/`、`\`）和 `..`。中文等 UTF-8 多字节字符（0x80-0xFF）允许通过。

- `query_records(class_str, keyword, results_buf, match_count)`：构建文件路径 `data/<class>.txt`，执行 `realpath()` 安全校验确保路径在 `data/` 目录内，逐行 `fgets()` 读取，用 `strstr()` 匹配 keyword。匹配行按制表符拆分为学号/姓名/性别三个字段，每个字段经 `html_escape()` 转义后拼接为 `<tr><td>...</td></tr>` HTML 表格行。文件不存在返回 -1（上层返回 404），无匹配返回 0（上层显示"未找到符合条件的记录"）。

- `handle_search_request(client_fd, method, query_str, &status_code, &mime_type, &body_bytes)`：`/search` 路由的主入口。GET 无参数 → 返回蓝色搜索表单（`#6f95c9` 配色，含"查询班级"和"关键字"输入框 + "提交"按钮）；GET/POST 有参数 → 解析查询串 → 校验 class 和 keyword → 调用 `query_records()` → 生成结果页 HTML → `send_response()` 分开发送响应头和响应体 → 记录系统日志。所有分支均通过统一的 `send_response()` 辅助函数发送响应，避免重复代码。

**POST 请求体处理**：在 `http_server.c` 的路由分发中，POST `/search` 先查找 `Content-Type` 请求头——缺失或非 `application/x-www-form-urlencoded` 返回 415 Unsupported Media Type；`Content-Length` 超过 4096 字节返回 413 Payload Too Large；`Content-Length` 缺失返回 400。V1.1/V1.2 已有的 `is_request_complete()` + `parse_http_request()` 机制确保请求体按 `Content-Length` 完整接收后才进入业务处理。

**动态 HTML 页面**：三个页面模板——搜索表单页（蓝色居中表单，指导书 `#6f95c9` 配色，表单 action 指向 `/search`，支持 GET 提交）、查询结果页（800px 白色容器，蓝色表头表格显示学号/姓名/性别，底部"返回查询"链接；无匹配时显示红色"未找到符合条件的记录"）、错误页（居中显示状态码和中文错误描述）。

**浏览器表单与 curl 命令相互验证**：`GET /search` 返回的 HTML 表单与指导书"前端设计"描述一致——蓝色矩形区域（550×260px，`#6f95c9` 背景），"查询班级"和"关键字"两行输入，白色输入框，透明白边框提交按钮。表单 method 为 GET，浏览器填写后提交 `/search?class=2011&keyword=男`，与 curl 命令等价。POST 版本的表单 method 为 POST，浏览器提交后请求体为 `class=2011&keyword=%E7%94%B7`，与 `curl -X POST -d` 等价。

对比验收标准全部检查点：编译与启动（`make clean && make` 无错误，`./mini_web_server serve-http 20` 成功监听 8080 端口）、GET /search 返回搜索表单（含"查询班级"和 `<form>` 元素）、GET /search?class=2011&keyword=%E7%94%B7 返回查询结果（匹配"张三""李四"，过滤"王五"）、GET 查询 keyword=女（匹配"王五"等女生）、POST /search 返回相同结果（与 GET 一致）、无匹配结果（显示"未找到符合条件的记录"）、class 格式错误（class=abc → 400 + "班级格式错误"）、class 不足 4 位（class=20 → 400）、数据文件不存在（class=9999 → 404 + "班级数据不存在"）、缺少 keyword 参数（→ 400）、POST 错误 Content-Type（text/plain → 415）、POST 请求体过大（>4096 字节 → 413）、PUT /search（→ 405 + Allow: GET, POST）、V1.2 静态资源兼容（GET / 首页正常 + CSS 正常）、V1.1 POST /echo 兼容（回显正常）、目录穿越防护（/../etc/passwd → 403）、查询处理器模块（7 个核心函数均存在）、epoll API 和连接清理（epoll_create1/epoll_ctl/epoll_wait + EPOLL_CTL_DEL + close）、跨班级查询（class=2012 正确返回 2012 班数据）、日志系统（访问日志含 /search 记录 + 系统日志含查询摘要），全部通过，自动测试脚本 **V1.3 38/38、V1.2 58/58、V1.1 38/38，合计 134 项全部通过**。

## 二、遇到的一个问题、原因和解决方案

**问题**：GET /search 返回的搜索表单中找不到 `<form>` 元素和"查询班级"文字，GET /search?class=2011&keyword=男 返回的结果页中找不到"张三""李四"等学生姓名，但错误页面（400/404）正常显示。

**原因**：`handle_search_request()` 中，HTTP 响应头和 HTML 响应体被一起放入 `char header[1024]` 缓冲区通过 `snprintf` 拼接后发送。搜索表单 HTML 和查询结果页 HTML 均超过 1024 字节（表单约 1446 字节，结果页约 1575 字节），导致 `snprintf` 截断输出。`snprintf` 返回的是"如果缓冲区足够大本应写入的字节数"，而非实际写入的字节数——后续 `send_all(client_fd, header, header_len)` 传入的是未截断的长度，虽未造成缓冲区溢出（`send_all` 只从 `header` 读取），但实际发送的内容只有前 1024 字节。客户端收到的 HTML 在 CSS 样式块末尾被截断，`<form>` 元素和表格行完全丢失。错误页面 HTML 较短（约 300 字节），未触发截断，因此正常显示。

**解决方案**：新增内部辅助函数 `send_response(int client_fd, int status_code, const char *reason, const char *mime, const char *body, int body_len)`，将响应头发送和响应体发送分离——先用 `snprintf` 将状态行 + Content-Type + Content-Length + Connection 格式化为简短头部（`char header[512]`，永远够用），`send_all()` 发送头部；再 `send_all()` 单独发送 HTML 响应体。两段发送后总计为完整响应。`handle_search_request()` 的六个分支全部改为调用 `send_response()`，消除重复的 `snprintf` + `send_all` 模式。修复后所有中文内容正常显示，浏览器和 curl 均能获取完整 HTML。

## 三、还存在什么不足，下一步如何改进

1. **数据文件查询为全表扫描（O(n)）**：当前 `query_records()` 逐行 `fgets()` + `strstr()` 扫描整个数据文件，1000 行文件需 1000 次字符串匹配。改进：为数据文件建立内存索引——服务器启动时加载所有 `data/*.txt` 到内存结构（如按班级分组的链表），查询时直接在内存中遍历，避免每次请求打开/关闭文件；定时检查文件 mtime 自动重载。

2. **不支持模糊查询和正则表达式**：当前仅支持 `strstr()` 子串精确匹配，用户输入"张"能匹配"张三"但输入"zhang"不能。改进：增加拼音首字母匹配（如"zs"匹配"张三"）、通配符（`*`、`?`）或正则表达式查询，提升查询灵活性。

3. **搜索表单仅支持 GET 提交**：当前 `generate_search_form_html()` 生成的表单 method 固定为 GET，虽可通过 curl 测试 POST，但浏览器用户无法直接使用 POST 方式。改进：在表单下方增加 GET/POST 切换按钮（radio button），或同时提供两个表单（一个 GET 一个 POST），让用户可以在浏览器中选择提交方式。

4. **无分页机制**：当查询结果超过数十条时全部显示在单页，页面过长。改进：增加 `page` 和 `page_size` 查询参数，每页显示固定条数（如 20 条），返回"上一页""下一页"导航链接，服务端只返回当前页的 HTML 表格行。

5. **错误页面信息过于简单**：当前 400 错误仅显示"参数格式错误"或"班级格式错误"，未告知用户正确的格式示例。改进：在错误页面中增加格式说明（如"班级编号为 4 位数字，例如 2011"）和返回搜索表单的链接，让用户可以直接修正后重新查询。

6. **未使用非阻塞 I/O 处理大请求体**：当前 POST 请求体在 `recv_buf` 中累积，但 send 文件内容时仍是阻塞模式。虽然 V1.3 的查询响应体较小（约 1.5KB），但如果后续增加图片生成等重操作，会阻塞事件循环。改进：将文件读取和 send 操作也纳入 epoll 管理，监听 EPOLLOUT 事件分块发送，与 EPOLLIN 交替推进多客户端并发。
