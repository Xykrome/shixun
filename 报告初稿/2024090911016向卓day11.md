# HTTP 协议与 Web 服务器 V2.0 训练报告

## 一、今天完成了什么

今天完成了 W3D1 的实训内容：**构建 HTTP 服务器 V2.0 —— 基于 epoll 的完整 HTTP 请求/响应处理与日志系统**。

### HTTP 协议理解

W3D1 的核心是理解 TCP 与 HTTP 的分工：**TCP 负责传输字节，HTTP 负责解释字节**。epoll 发现哪个 fd 有事件（W2D5），HTTP 协议负责解释 fd 中收到的字节流并生成规范响应（W3D1）。一次 HTTP 通信包含请求和响应：浏览器建立 TCP 连接 → 发送 HTTP 请求 → 服务器解析请求 → 返回 HTTP 响应 → 浏览器渲染页面。

HTTP 请求由四部分组成：请求行（METHOD SP PATH SP VERSION CRLF）、请求头部（Key: Value CRLF）、空行（CRLF）、请求体。HTTP 响应同样四部分：状态行（VERSION SP STATUS SP REASON CRLF）、响应头部、空行、响应体。**关键认知**：TCP 只提供字节流，一次 recv() 不等于一个完整的 HTTP 请求——请求可能分多次到达，需要通过查找 `\r\n\r\n` 判断请求头完整性，再通过 Content-Length 判断请求体是否完整接收。

### 实训任务：HTTP Server V2.0

在既有 `mini_webserver/` 项目中新增 `include/http_server.h`、`src/http_server.c`、`include/http_parser.h`、`src/http_parser.c`，大幅增强 `src/log.c` / `include/log.h`，修改 `src/main.c`（新增 `serve-http <max_requests>` 命令分支）、`Makefile`（新增编译目标和 `make test10`），新建 `tests/test_day11.sh`。

**核心架构**：`socket → bind → listen → epoll_create1 → epoll_ctl(ADD listen_fd) → while (request_count < max_requests) { epoll_wait → 就绪事件 → accept 或 recv → 追加到客户端接收缓冲区 → 判断请求完整性(查找\r\n\r\n + Content-Length) → HTTP 解析(请求行/请求头/请求体) → 路由分发 → 生成响应 → send → 访问日志 + 系统日志 → epoll_ctl(DEL) + close → request_count++ } → 正常退出`。

**与 W2D5 V1.0 的关键区别**：

| 维度 | V1.0 (W2D5) | V2.0 (W3D1) |
|------|------------|------------|
| HTTP 解析 | `sscanf` 一次提取，不处理请求体 | 逐字节追加缓冲区，查找 `\r\n\r\n` 分隔符，完整解析请求行/头/体 |
| 请求完整性 | 假设一次 recv() 即是完整请求 | `is_request_complete()` 判断：查找 `\r\n\r\n` + Content-Length 验证 |
| POST 支持 | 不支持 | 支持 POST /echo，正确解析请求体 |
| 消息分帧 | 未考虑粘包/半包 | 缓冲区追加模式，支持任意分片到达 |
| 日志系统 | 单一日志文件，仅 INFO/ERROR | 系统日志 + 访问日志分离，DEBUG/INFO/WARNING/ERROR 四级 |
| 请求计数 | 每个连接 1 请求 | 同 V1.0，也是每连接 1 请求 |

**HTTP 解析器**（`http_parser.h/c`）：
- `find_header_end()`：在缓冲区中查找 `\r\n\r\n`（请求头结束标记），返回请求体开始偏移
- `get_content_length()`：从请求头中解析 Content-Length 值
- `is_request_complete()`：同时检查请求头完整性和请求体完整性
- `parse_http_request()`：提取 method/path/version/headers/body 到 `http_request_t` 结构体

**路由规则**（本次实训必做范围）：
- `GET /` → HTTP/1.1 200 OK + HTML 页面（Content-Type: text/html）
- `GET /missing` → HTTP/1.1 404 Not Found + HTML 错误页
- `POST /echo` → HTTP/1.1 200 OK + 回显请求体（Content-Type: text/plain）
- 其他 HTTP 方法 → HTTP/1.1 405 Method Not Allowed
- 其他路径 → HTTP/1.1 404 Not Found

**响应格式**（本次实训的重点要求）：
```http
HTTP/1.1 200 OK\r\n
Content-Type: text/html; charset=utf-8\r\n
Content-Length: 197\r\n
Connection: close\r\n
\r\n
<!DOCTYPE html>...<h1>Hello, HTTP!</h1>...
```
每行以 `\r\n` 结束，空行分隔响应头和响应体，Content-Length 精确计算响应体的字节数。

**日志系统增强**（`log.h/c` 重写）：

系统日志格式：`YYYY-MM-DD HH:MM:SS [LEVEL] 消息`
访问日志格式：`YYYY-MM-DD HH:MM:SS IP METHOD PATH HTTP版本 状态码 响应字节数`

两个日志分别写入不同文件（`logs/system.log` 和 `logs/access.log`），支持四级日志（DEBUG、INFO、WARNING、ERROR）。每个 HTTP 请求处理完毕后，同时写入访问日志（记录客户端 IP、方法、路径、状态码）和系统日志（记录路由分发结果）。

对比验收标准的全部检查点：编译与启动（`./mini_web_server serve-http 8`）、GET / 路由（curl → 200 OK + HTML + Content-Type + Content-Length）、GET /missing 路由（curl → 404 Not Found + 服务器不崩溃）、POST /echo 路由（curl -X POST -d "hello HTTP" → 200 OK + "Echo: hello HTTP"）、405 方法检查（PUT → 405 Method Not Allowed）、多请求处理（连续 3 个请求全部 200 OK）、请求数量控制（8 个请求后正常退出）、epoll API 检查（epoll_create1/epoll_ctl/epoll_wait/LT+EPOLLIN）、技术限制（无 select/多线程/多进程）、连接清理（EPOLL_CTL_DEL + close）、HTTP 解析器检查（`\r\n\r\n` + Content-Length）、日志系统（系统日志 + 访问日志分离，包含时间/IP/方法/路径/状态码，支持四级日志），全部通过，自动测试脚本 **38/38 项通过**。

## 二、遇到的一个问题、原因和解决方案

**问题**：编译时出现 `conflicting types for 'client_info_t'` 错误，以及 `log_init` 参数数量不匹配错误。

**原因**：

1. `client_info_t` 结构体在 `epoll_server.h`（V1.0）和 `http_server.h`（V2.0）中重复定义。`main.c` 同时 include 了两个头文件，导致类型冲突。两个结构的字段完全相同，但分属不同的独立模块。

2. `log_init` 函数签名从 W2D5 的 `log_init(const char *path)`（单一日志文件）改为 W3D1 的 `log_init(const char *sys_log_path, const char *access_log_path)`（系统日志 + 访问日志分离）。但 `main.c` 中 `serve-epoll` 分支和 Web 服务器模式分支仍使用旧的单参数调用方式。

**解决方案**：

1. 在两个头文件中添加 `#ifndef CLIENT_INFO_T_DEFINED` / `#define CLIENT_INFO_T_DEFINED` 条件编译保护，无论 include 顺序如何，`client_info_t` 只定义一次。

2. 将旧调用 `log_init("logs/server.log")` 改为 `log_init("logs/server.log", NULL)`（第二个参数传 NULL，log.c 内部支持 NULL 路径），`log_init(config.log_path)` 改为 `log_init(config.log_path, NULL)`。V2.0 使用完整的双参数调用 `log_init("logs/system.log", "logs/access.log")`。

## 三、还存在什么不足，下一步如何改进

1. **不支持 HTTP Keep-Alive（持久连接）**：当前每个连接处理一个请求后立即关闭（`Connection: close`），浏览器访问时每个资源都需要重新建立 TCP 连接。改进：响应头设置 `Connection: keep-alive`，处理完一个请求后不关闭 fd，继续在 epoll 中监听同一 fd 上的后续请求，直到超时或客户端主动关闭。

2. **请求体大小受限**：当前请求体最大为 8192 字节（HTTP_BODY_MAX），超过此大小的 POST 请求会被截断。改进：支持动态分配更大的请求体缓冲区，或使用分块读取的方式处理大文件上传。

3. **不支持静态文件服务**：虽然是 HTTP 服务器，但当前只处理硬编码的三个路由。`www/` 目录下已有 `index.html` 文件，但服务器不会自动读取文件系统返回静态资源。改进：增加文件系统路由处理器，将 URL path 映射到 `www/` 目录，根据文件扩展名设置正确的 Content-Type（从 `/etc/mime.types` 或硬编码 MIME 表获取），实现真正的静态文件 Web 服务。

4. **路由机制硬编码**：当前路由逻辑直接写在 `generate_response()` 函数中，每新增一个路由都需要修改该函数。改进：引入路由注册机制（路由表），每个路由项包含 method、path、handler 函数指针，支持前缀匹配和精确匹配，方便扩展。

5. **日志文件无轮转机制**：日志文件无限追加，长时间运行会变得非常大。改进：增加日志轮转（log rotation），按文件大小或日期自动切换新日志文件，保留最近 N 个历史日志。

6. **不支持 HTTP/1.1 Host 头验证**：HTTP/1.1 规范要求请求必须包含 Host 头，当前服务器未验证。改进：解析 Host 头，支持基于域名的虚拟主机（Virtual Host），不同域名映射到不同的 `www/` 子目录。

7. **ET 模式和 EPOLLONESHOT 未使用**：当前使用默认 LT 模式。W3D1 课件提到 ET 和 EPOLLONESHOT 作为扩展方向。改进：添加 `--et` 命令行参数，启用边缘触发模式时配合非阻塞 I/O（`fcntl(O_NONBLOCK)`）和循环 recv/send 直到 EAGAIN，减少 epoll_wait 唤醒次数，提升高并发场景下的性能。
