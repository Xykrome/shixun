# Webserver V1.1 训练报告

## 一、今天完成了什么

今天完成了 W3D1 的实训内容：**构建 HTTP 服务器 V1.1 —— 基于 epoll 的完整 HTTP 请求解析、路由分发与日志系统**。

W3D1 的核心是理解 **TCP 与 HTTP 的分工**：epoll 负责发现哪个 fd 有事件（W2D5），HTTP 协议负责解释 fd 中收到的字节流并生成规范响应（W3D1）。TCP 只提供字节流，**一次 recv() 不等于一个完整的 HTTP 请求**——请求可能分多次到达，也可能多个请求一起到达。因此需要按客户端维护接收缓冲区，每次 recv 后追加数据，查找 `\r\n\r\n` 判断请求头是否完整，再根据 Content-Length 判断请求体是否接收完毕，完整后才开始解析。

在既有 `mini_webserver/` 项目中新增 `include/http_parser.h`、`src/http_parser.c`（HTTP 请求解析器）、`include/http_server.h`、`src/http_server.c`（epoll HTTP 服务器），重写 `include/log.h`、`src/log.c`（增强日志系统），修改 `src/main.c`（新增 `serve-http <max_requests>` 命令分支）和 `Makefile`（新增 `http_parser.c`、`http_server.c` 编译目标与 `make test10` 测试目标），新建 `tests/test_day11.sh`。

**核心架构**：`socket → bind → listen → epoll_create1(0) → epoll_ctl(ADD listen_fd) → while (request_count < max_requests) { epoll_wait() → 如果 listen_fd 就绪则 accept() 加入客户端列表并 epoll_ctl(ADD)；否则 recv() → 追加到该客户端的接收缓冲区 → is_request_complete() 判断请求是否完整（查找 \r\n\r\n + 验证 Content-Length）→ 不完整则继续等待下一次 recv；完整则 parse_http_request() 解析请求行/请求头/请求体 → generate_response() 路由分发 → send() 发送 HTTP 响应 → access_log() + log_info() 记录日志 → epoll_ctl(DEL) + close(client_fd) → request_count++ } → 达到 max_requests 后正常退出`。

**与 W2D5 V1.0 的关键对比**：

| 维度 | V1.0 (W2D5) | V1.1 (W3D1) |
|------|------------|------------|
| HTTP 接收 | 一次 recv() 后直接用 sscanf 解析，假设一次收到完整请求 | 按客户端维护接收缓冲区，每次 recv() 追加，查找 `\r\n\r\n` 分隔符 |
| 请求完整性判断 | 无 | `is_request_complete()`：先找 `\r\n\r\n`，再根据 Content-Length 验证请求体 |
| 请求解析 | sscanf 提取 method/path/protocol，不解析请求头和请求体 | 完整解析：请求行、多个请求头键值对、Content-Length、请求体 |
| POST 支持 | 不支持（仅处理 GET） | 支持 POST /echo，正确解析请求体并回显 |
| 路由 | /hello、/users/\<name\>、其他→404 | GET /→200+HTML、GET /missing→404、POST /echo→200+回显 |
| 响应格式 | Content-Type: text/plain | Content-Type: text/html（带 charset=utf-8）+ Content-Length + Connection |
| 日志 | 单一日志文件，仅 INFO/ERROR 两级 | 系统日志 + 访问日志分离，DEBUG/INFO/WARNING/ERROR 四级 |

**HTTP 解析器**（`http_parser.h/c`）实现了四个函数：
- `find_header_end(buf, len)`：在字节流中查找 `\r\n\r\n`，返回请求体起始偏移，未找到返回 -1
- `get_content_length(header_part, len)`：从请求头部分解析 Content-Length 头的值
- `is_request_complete(buf, len)`：先查找 `\r\n\r\n`，有 Content-Length 时验证请求体是否完整
- `parse_http_request(buf, len, &req)`：解析完整的 HTTP 请求，提取 method、path、version、headers 数组（最多 16 个）、body、body_len，存入 `http_request_t` 结构体

**路由规则**（验收标准要求的必做范围）：
- `GET /` → HTTP/1.1 200 OK，Content-Type: text/html，返回包含 `<h1>Hello, HTTP!</h1>` 的 HTML 页面
- `GET /missing` → HTTP/1.1 404 Not Found，Content-Type: text/html，返回 404 错误页面
- `POST /echo` → HTTP/1.1 200 OK，Content-Type: text/plain，响应体为 `Echo: <请求体内容>`
- 其他 HTTP 方法（如 PUT）→ HTTP/1.1 405 Method Not Allowed，Allow: GET, POST
- 其他 GET/POST 路径 → HTTP/1.1 404 Not Found

**响应格式**严格遵循 HTTP/1.1 规范：每行以 `\r\n` 结束，空行（单独的 `\r\n`）分隔响应头和响应体，Content-Length 精确等于响应体的字节数，Connection: close 表示处理完毕后关闭连接。

**日志系统增强**（`log.h/c` 重写）：系统日志和访问日志分别写入 `logs/system.log` 和 `logs/access.log` 两个文件。系统日志格式为 `YYYY-MM-DD HH:MM:SS [LEVEL] 消息`，支持 DEBUG、INFO、WARNING、ERROR 四级，记录服务器启动、客户端连接/断开、请求处理结果、错误等事件。访问日志格式为 `YYYY-MM-DD HH:MM:SS IP METHOD PATH HTTP版本 状态码 响应字节数`，每个 HTTP 请求处理完毕后写入一行。两个日志文件均以追加模式打开，每次写入后 `fflush` 确保日志实时落盘。

对比验收标准的全部检查点：编译与启动（`make clean && make` 无错误，`./mini_web_server serve-http 8` 成功监听 8080 端口）、GET / 路由（curl -i → 200 OK + Content-Type: text/html + `<h1>Hello, HTTP!</h1>`）、GET /missing 路由（curl -i → 404 Not Found + 服务器不崩溃继续处理后续请求）、POST /echo 路由（curl -i -X POST -d "hello HTTP" → 200 OK + Echo: hello HTTP）、响应格式（每行 `\r\n` 结束，Content-Type 和 Content-Length 正确设置）、多请求处理（连续 3 个不同类型请求全部正常响应，服务器不中断）、请求数量控制（处理完 max_requests 个请求后正常退出）、连接清理（每次响应完成后执行 epoll_ctl EPOLL_CTL_DEL + close）、epoll API（epoll_create1/epoll_ctl/epoll_wait/LT+EPOLLIN 正确使用）、技术限制（未使用 select/多线程/多进程/线程池）、日志分离（system.log 和 access.log 分别写入）、访问日志内容（时间 + 客户端 IP + 方法 + URL + HTTP 版本 + 状态码 + 响应字节数）、系统日志内容（时间 + 级别 + 事件描述）、日志级别（代码支持 log_debug/log_info/log_warning/log_error 四级）、三类请求均有访问记录（GET / 200、GET /missing 404、POST /echo 200），全部通过，自动测试脚本 **38/38 项通过**。

## 二、遇到的一个问题、原因和解决方案

**问题**：编译时出现两个错误：(1) `conflicting types for 'client_info_t'` 类型冲突；(2) `too few arguments to function 'log_init'` 参数数量不匹配。

**原因**：

第一个错误：`client_info_t` 结构体在 `epoll_server.h`（W2D5 V1.0）和 `http_server.h`（W3D1 V1.1）中各自定义了一份。两个结构体的字段完全相同（fd、ip、port、recv_buf、buf_len），但 `main.c` 同时 include 了两个头文件，编译器看到同一个类型被定义了两次，报类型冲突错误。两个模块分别独立开发，各自需要客户端信息结构体，没有抽取公共定义。

第二个错误：`log_init` 的函数签名在 W3D1 中被改为 `log_init(const char *sys_log_path, const char *access_log_path)`（系统日志 + 访问日志两个路径），但 `main.c` 中 V1.0 的 `serve-epoll` 分支和 V0.4-V0.8 的 Web 服务器模式分支仍使用旧的单参数调用：`log_init("logs/server.log")` 和 `log_init(config.log_path)`。编译器提示参数不足。

**解决方案**：

1. 在两个头文件中分别添加 `#ifndef CLIENT_INFO_T_DEFINED` / `#define CLIENT_INFO_T_DEFINED` 条件编译保护。无论 `main.c` 以什么顺序 include 头文件，`client_info_t` 只会在第一个被 include 的头文件中定义一次，后续头文件中的定义被 `#ifndef` 跳过。这种方案保持了两个模块的独立性，不需要引入额外的公共头文件。

2. 将旧代码中的单参数调用改为双参数：`log_init("logs/server.log", NULL)` 和 `log_init(config.log_path, NULL)`。传入 NULL 表示不需要访问日志文件，`log_init` 内部对 NULL 路径做检查后跳过对应文件的打开。V1.1 的 `serve-http` 分支使用完整的双参数调用 `log_init("logs/system.log", "logs/access.log")` 同时初始化两个日志文件。

## 三、还存在什么不足，下一步如何改进

1. **不支持 HTTP Keep-Alive（持久连接）**：当前每个连接处理一个请求后立即关闭（`Connection: close`），浏览器访问时每个资源（HTML/CSS/JS/图片）都需要重新建立 TCP 连接，增加延迟和资源消耗。改进：响应头保留 `Connection: keep-alive`，处理完一个请求后不关闭 fd，继续在 epoll 中监听同一 fd 上的后续请求，配合超时机制（如 60 秒无活动自动关闭），实现 HTTP/1.1 持久连接，同一连接上复用多次请求-响应。

2. **请求体大小固定受限**：当前请求体最大为 8192 字节（HTTP_BODY_MAX），超过此大小的 POST 请求会被截断，不适合文件上传等场景。改进：支持动态分配请求体缓冲区，或在解析到 Content-Length 较大时使用临时文件存储请求体，处理完后再删除。

3. **不支持静态文件服务**：当前仅处理三个硬编码路由（GET /、GET /missing、POST /echo），`www/` 目录下已有 `index.html` 但服务器不会自动读取文件系统返回。改进：增加文件系统路由处理器，将 URL path 映射到 `www/` 目录下的实际文件，根据文件扩展名设置正确的 Content-Type（如 `text/html`、`text/css`、`image/png`），文件不存在时返回 404。

4. **路由机制硬编码在 generate_response() 中**：当前每增加一个路由都需要修改 `generate_response()` 函数，路由和响应生成逻辑耦合在一起。改进：引入路由表注册机制——每个路由项包含 method、path、handler 函数指针，启动时注册路由，请求到来时查表分发，分离路由注册与响应生成。

5. **日志文件无轮转机制**：日志以追加模式无限写入，长时间运行后日志文件会变得非常大，查找困难且占用磁盘空间。改进：增加日志轮转——按文件大小（如 10MB）或按日期自动切换新日志文件，旧文件重命名为 `system.log.1`、`system.log.2`，保留最近 N 个历史日志。

6. **未使用 ET 模式和 EPOLLONESHOT**：当前两个项目均使用默认的 LT（电平触发）模式。指导书提到 ET 模式和 EPOLLONESHOT 作为扩展方向。改进：添加命令行参数 `--et` 启用边缘触发模式（`EPOLLET`），配合非阻塞 I/O（`fcntl(fd, F_SETFL, O_NONBLOCK)`），在每次可读事件到来后循环 recv 直到返回 EAGAIN，减少 epoll_wait 唤醒次数；对后续的多线程版本增加 `EPOLLONESHOT`，确保同一时刻只有一个线程处理一个 fd，避免竞态条件。
