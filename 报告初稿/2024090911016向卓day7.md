# 实验目的与内容

## 1.1 实验目的

1. 掌握多进程并发服务器的基础架构，理解 `fork()` 系统调用为每个客户端连接创建子进程的工作模型，领会父进程和子进程在文件描述符管理上的分工（父进程 `close(conn_fd)`、子进程 `close(listen_fd)`）。

2. 掌握 `SIGCHLD` 信号的产生机制和处理方法，理解僵尸进程（zombie）的产生原因及危害，学会使用 `signal()` 注册信号处理函数、使用 `waitpid(-1, &stat, WNOHANG)` 非阻塞回收子进程资源，避免僵尸进程占用内核空间。

3. 掌握 `accept()` 被信号打断的处理方式，理解慢系统调用（slow system call）被信号中断时返回 `EINTR` 错误的原理，学会在代码中判断 `errno == EINTR` 并 `continue` 重新 `accept()` 的正确做法。

4. 掌握 `SIGPIPE` 信号的触发场景和处理策略——当客户端异常断开连接后服务器仍向已关闭的套接字写入数据时，内核发送 `SIGPIPE` 信号，默认行为是终止进程。学会使用 `signal(SIGPIPE, SIG_IGN)` 忽略此信号，使 `send`/`write` 返回 `EPIPE` 错误而非终止整个服务器。

5. 通过实现多客户 TCP 文本聊天程序，理解多进程并发服务器中三个进程（客户端、服务器父进程、服务器子进程）各自阻塞于不同系统调用（`fgets`/`read`，`accept`，`recv`）的并发执行状态，掌握 TCP 连接从建立到终止的完整生命周期在并发场景下的表现。

6. 在 mini_webserver 中实现多进程并发 TCP 网络通信模块（V0.7），从 V0.6 的单连接服务器升级为可同时处理多个 curl 客户端请求的并发服务器。理解 `fork`、`SIGCHLD`、`waitpid`、`EINTR`、`SIGPIPE` 在实际 Web 服务器中的综合应用。

# 实验原理

## 2.1 多进程并发服务器基本架构

多进程并发服务器是 Linux 网络编程中最经典的并发模型之一。其核心思想是：父进程负责监听和接受客户端连接（`accept`），每接受一个连接就调用 `fork()` 创建一个子进程，由子进程独立处理该客户端的请求，父进程则立即返回继续监听新的连接请求。这种模型将一个复杂的"同时服务多个客户端"问题分解为多个独立的"服务单个客户端"问题，每个子进程的处理逻辑与单连接服务器完全相同。

基本架构伪代码（W2D2 Slide 2）：

```
socket() → bind() → listen()
for (;;) {
    connfd = accept(listenfd, ...);
    if (fork() == 0) {          // 子进程
        close(listenfd);         // 关闭监听套接字
        doit(connfd);            // 处理客户端请求
        close(connfd);           // 关闭连接套接字
        exit(0);                 // 子进程退出
    }
    close(connfd);               // 父进程关闭连接套接字
}
```

`fork()` 后父进程和子进程各自持有 `listen_fd` 和 `conn_fd` 的副本——子进程应立即 `close(listen_fd)` 释放监听描述符，父进程应立即 `close(conn_fd)` 释放连接描述符。如果父进程不关闭 `conn_fd`，子进程关闭后该套接字的引用计数不会归零，TCP FIN 不会发出，客户端连接无法正常终止。

## 2.2 fork() 系统调用

`fork()` 是 Unix/Linux 中创建新进程的唯一方法。调用 `fork()` 后，内核创建一个与父进程几乎完全相同的新进程（子进程），包括：代码段、数据段、堆栈、打开的文件描述符表、信号处理方式、当前工作目录等。`fork()` 的返回值是区分父子进程的关键——在父进程中返回子进程的 PID（>0），在子进程中返回 0，创建失败返回 -1。父子进程的调度顺序由内核决定，无法预知谁先运行。在多进程并发服务器中，`fork()` 使得服务器可以为每个客户端创建独立的执行上下文，多个子进程并发运行，互不干扰。

## 2.3 SIGCHLD 信号与僵尸进程

当一个进程终止时，内核会向其父进程发送 `SIGCHLD` 信号（信号编号 17）。该信号的默认处理方式是忽略（`SIG_DFL`）。然而，"忽略"仅指父进程可以不处理该信号，并不意味着子进程的资源被自动回收——子进程终止后，内核仍保留其进程描述符（task_struct）中的部分信息（PID、终止状态、CPU 时间等），直到父进程调用 `wait`/`waitpid` 取回这些信息。处于这种"已终止但未被父进程回收"状态的进程称为僵尸进程（zombie），在 `ps` 输出中显示为 Z 状态。

僵尸进程的危害在于：它们占用内核进程表项（PID 资源），如果父进程长期不回收，僵尸进程会不断积累，最终耗尽系统的 PID 资源（默认上限 32768），导致无法创建新进程。因此，"无论何时创建子进程都必须等待（wait）它们，以免变成僵尸进程"（W2D2 Slide 12）。处理方式是注册 `SIGCHLD` 信号处理函数，在处理函数中调用 `wait`/`waitpid` 回收子进程。如果一个进程终止时其父进程已经不存在，则该进程的父进程会被设为 `init` 进程（PID=1），由 `init` 负责 `wait` 回收。

## 2.4 wait() 与 waitpid() 函数

**wait() 函数原型**：`pid_t wait(int *statloc)`。调用 `wait` 的进程将阻塞直到任意一个子进程终止，返回终止子进程的 PID，并通过 `statloc` 返回其终止状态。`wait()` 的问题是：如果有多个子进程几乎同时终止，`wait` 每次只能回收一个，且调用者会阻塞。

**waitpid() 函数原型**：`pid_t waitpid(pid_t pid, int *statloc, int options)`。相比 `wait` 提供了更精细的控制：`pid` 参数指定要等待的子进程 PID（-1 表示等待任意子进程，等效于 `wait`）；`options` 参数指定附加选项，最常用的是 `WNOHANG`，它通知内核在没有已终止子进程时不要阻塞，立即返回 0。在信号处理函数中必须使用 `waitpid + WNOHANG` 的组合，因为在信号处理函数中不能长时间阻塞。

此外，由于 **Unix 信号不排队**——多个子进程几乎同时终止时可能只触发一次 `SIGCHLD`——信号处理函数中应使用 `while` 循环不断调用 `waitpid(-1, &stat, WNOHANG)`，直到返回 ≤ 0 为止，确保一次性回收所有已终止子进程。这是 W2D2 强调的关键实现细节。

## 2.5 accept() 被信号打断（EINTR）

`accept()` 是典型的"慢系统调用"（slow system call）——调用者可能永远阻塞，直到有客户端连接到达。当父进程阻塞于 `accept()` 时，如果收到 `SIGCHLD` 信号，内核会中断 `accept()` 调用，使其返回 -1 并设置 `errno = EINTR`（Interrupted system call）。如果不处理这种情况，程序将把 `EINTR` 当作致命错误并中止 accept 循环，导致服务器停止接受新连接。

正确的处理方式（W2D2 Slide 14）：

```c
conn_fd = accept(listen_fd, ...);
if (conn_fd < 0) {
    if (errno == EINTR) {
        continue;          /* 被信号打断，重新 accept */
    }
    perror("accept error");
    break;
}
```

## 2.6 SIGPIPE 信号处理

`SIGPIPE` 信号的触发场景（W2D2 Slide 20）：当一个进程向已收到 RST（复位）的套接字进行写操作时，内核给该进程发送 `SIGPIPE` 信号。这种情况在多进程服务器中很容易发生——客户端在服务器子进程发送响应之前就异常断开连接（如 curl 进程被 Ctrl+C 终止），子进程仍然调用 `send()` 写数据到已关闭的连接，触发 `SIGPIPE`。

`SIGPIPE` 的默认行为是终止进程。在多进程服务器中，处理方式是将其设为 `SIG_IGN`（忽略），此时 `send`/`write` 操作将返回 -1 并设置 `errno = EPIPE`，进程不会被终止，可以优雅地处理这个错误（记录日志、关闭连接、退出子进程）。

## 2.7 多进程 TCP 服务器的 C/S 工作流程

服务器端流程：`socket() → setsockopt(SO_REUSEADDR) → bind() → listen() → signal(SIGCHLD, handler) + signal(SIGPIPE, SIG_IGN) → 循环{accept()（检查EINTR）→ fork() → 子进程{close(listen_fd) → recv() → handle_request_string() → send() → close(conn_fd) → exit(0)} → 父进程{close(conn_fd) → continue}}`。

三个进程的并发执行状态（W2D2 Slide 7）：客户端阻塞于 `fgets`/`read` 等待服务器响应；服务器父进程阻塞于 `accept` 等待新客户端连接；服务器子进程阻塞于 `recv` 等待当前客户端发送 HTTP 请求。三个进程都在睡眠（阻塞），直到某个事件发生（新连接到达、数据到达）将对应进程唤醒。这种并发模型充分利用了 Unix 的进程调度机制——当一个进程阻塞时，内核调度其他就绪进程运行，从而实现"同时"服务多个客户端。

### 实验结果

> **分析**：V0.7 的实现完成了 mini_webserver 从"单连接"到"多连接并发"的关键跨越。V0.6 已经解决了"网络通信"的问题（真实的 TCP 连接取代文件模拟），但每次只能处理一个客户端。V0.7 通过 `fork()` 引入了进程级并发，使得多个 curl 客户端可以"同时"获得服务。从架构角度看，V0.7 遵循了 Unix 编程中经典的"一连接一进程"模型（process-per-connection），这也是早期 Apache HTTP Server（prefork MPM）的基础架构。每个子进程独立运行、崩溃不影响其他进程——这是多进程模型相比多线程模型（V0.5）在隔离性上的优势。其局限性在于 `fork()` 开销较大，大量并发连接时进程切换和内存消耗显著——后续可引入 I/O 多路复用（select/poll/epoll）实现单进程内的高效并发。

# 心得体会与建议

通过本次实验，我从理论到实践完整地学习了 Linux 下的多进程并发网络编程。W2D2 的课程内容沿着"多进程并发架构 → fork 系统调用 → 信号处理（SIGCHLD/SIGPIPE）→ 异常处理（EINTR/僵尸进程）→ 代码实战"的路径展开，每一个知识点都直击并发编程中的实际痛点——僵尸进程、慢系统调用中断、客户端异常断开——这些都不是课堂上能凭空想象的，只有亲手编写代码、亲眼看到 bug 现象、亲手修复后才能真正理解。

训练1 的多客户 TCP 聊天程序让我第一次体会到并发编程的威力——当打开三个终端，两个客户端同时向服务器发消息，各自收到独立的回射时，`fork()` 创造的"多个执行流"才不再是抽象概念。特别是观察服务器终端输出的 PID——同一个通信循环，运行在不同的进程上下文中，父进程的"子进程 xxx 已终止（回收成功）"消息穿插在 accept 新连接的信息之间——这种输出交错的模式正是并发执行的直观证据。

Webserver V0.7 的实现是整个 mini_webserver 项目演进中的重要里程碑。从架构演进的角度来看：V0.4（多进程文件模式）在进程模型上做了铺垫，V0.5（多线程文件模式）引入了线程并发，V0.6（TCP 单连接）实现了真正的网络通信，V0.7（TCP 多进程并发）将"并发"和"网络"首次结合在一起。每一步都在前一步的基础上增加一个维度，每一步的业务逻辑（handle_request/handle_request_string）都被下一版本完全复用——这是渐进式架构演进的优秀实践。

**在实现过程中遇到的几个关键点和反思：**

**(1) 僵尸进程的隐蔽性**——在开发初期，我忘记了注册 SIGCHLD 处理函数。程序表面上一切正常：客户端能连接、能收到响应、能正常退出。但运行 `make test7` 几次后，`ps aux` 显示大量 `<defunct>` 进程，PID 资源逐渐耗尽。加上 SIGCHLD 处理函数后僵尸进程消失。这说明并发编程中的资源管理不能依赖"程序看起来正常"的感觉——僵尸进程不产生 core dump，不触发段错误，只有通过工具（ps/top）或 PID 耗尽后的 "fork failed" 错误才能暴露。

**(2) accept() EINTR 的疏漏**——在第一个版本中，我没有在 `accept()` 返回 -1 后检查 `errno == EINTR`。测试时刚好没有触发这个问题（测试顺序使得信号在 accept 返回之后才到达），但代码是不完备的。加上 EINTR 处理后，服务器在被信号中断时能正确恢复——这种"防御性编程"的思想在并发编程中尤为重要：不能依赖"刚好不会触发"的假设。

**(3) SIGPIPE 的代价**——V0.6 单连接服务器中，客户端断开后服务器即退出，SIGPIPE 不会造成实际损害。但在 V0.7 多进程场景下，如果不对 SIGPIPE 做处理，某个子进程因客户端异常断开而收到 SIGPIPE 默认处理（进程终止），虽然子进程终止不会影响其他子进程，但频繁的异常终止会掩盖真正的问题。将 SIGPIPE 设为 `SIG_IGN` 是最简单有效的防御策略——将"进程崩溃"降级为"send 返回 EPIPE 错误"，后者可以被优雅处理。

**(4) 父子进程 fd 管理的对等性**——`fork()` 后父子进程各自持有 `listen_fd` 和 `conn_fd` 的副本，需要分别关闭不需要的文件描述符。父进程如果不 `close(conn_fd)`，子进程 `close(conn_fd)` 后套接字的引用计数从 2 减为 1，TCP 协议栈不会发送 FIN 段，客户端连接陷入"半关闭"状态——curl 可能永远阻塞。理解并正确处理这一双向 fd 管理是掌握 fork() 网络编程的关键。

**建议：**(1) 本版本的"一连接一进程"模型在并发量不大时（如测试中的 5 个并发 curl）运行良好，但生产环境面对数千并发连接时，fork() 的开销和每个子进程的内存占用将成为瓶颈。后续可引入 I/O 多路复用（select/poll/epoll）在单进程/少量进程中管理大量连接。(2) 可以引入进程池（prefork）模型——预先创建一定数量的子进程，避免每次连接都 fork() 的开销——这是 Apache HTTP Server prefork MPM 的核心思想。(3) 当前 `handle_request_string` 假设一次 `recv` 就能获取完整 HTTP 请求，后续应为 `recv` 添加循环读取和 `\r\n\r\n` 边界检测。(4) `MAX_CLIENTS` 硬编码为 10 是为了方便自动测试——生产环境中应通过配置文件或命令行参数指定。
