# Webserver V1.4 训练报告

## 一、今天完成了什么

今天完成了 W3D4 的实训内容：**将 V1.3 升级为配置驱动的 V1.4 —— 服务器启动时从 `server.json` 读取参数和路由，校验后构建路由表，请求到达时按 `method + path` 查找路由并调用 handler，无匹配则回退到静态文件服务**。

W3D3 解决了"怎样处理动态请求"，W3D4 解决的是"从哪里获取配置和路由"——不再把 host/port/document_root/log/routes 硬编码在代码中，而是通过 JSON 配置文件描述，修改配置后重启生效，无需重新编译。核心新增模块引入 **cJSON**（v1.7.19，MIT 协议，业界成熟 JSON 库）替代自写解析器；`config.h/c` 完备重写，支持 JSON 加载与 11 项字段校验；`http_server.c` 的路由分发从硬编码 if/else 重构为配置驱动的路由表查找。

**新增文件**：`include/cJSON.h`、`src/cJSON.c`（cJSON v1.7.19 成熟 JSON 库，MIT 协议）、`config/server.json`（服务器配置文件，描述 host/port/document_root/log/routes）、`tests/test_day14.sh`（32 项自动化测试）。

**修改文件**：`include/config.h`（`server_config_t` 新增 `document_root`、`log_file`、`log_level`、`routes[]` 字段，同时保留 V0.x 兼容字段 `root`、`log_path`、`server_name`）、`src/config.c`（完备重写 —— `load_json_config()` 用 cJSON 解析 + 11 项校验 + 填充配置；`load_legacy_config()` 保留 INI 解析用于 V0.x 兼容；`print_config()` 适配新字段）、`include/static_handler.h`（新增 `set_document_root()` / `get_document_root()` API）、`src/static_handler.c`（以模块级变量 `g_document_root` 替代硬编码 `WWW_ROOT` 宏，默认 "www" 保持向后兼容）、`include/http_server.h`（`http_server_run()` 签名从 `(int port, int max)` 改为 `(const server_config_t *config, int max)`；新增 `Handler` 函数指针类型和 `find_handler_by_name()` API）、`src/http_server.c`（约 600 行重构 —— 从配置构建运行时路由表、`find_route()` 查找、`build_allow_header()` 动态生成 Allow 头、`extract_path_no_query()` 去查询串用于路由匹配、handler 包装函数 `search_get_handler / search_post_handler / echo_post_handler` 实现统一签名、dispatch 逻辑：路由匹配→handler / path 存在但 method 错→405 / GET 回退静态文件 / 非 GET 无路由→405）、`src/main.c`（新增 V1.4 CLI：`./mini_web_server <file.json> [max_requests]`；`serve-http` 模式内部转为 config+route-table 运行；V0.x 模式使用 `load_legacy_config()`）、`Makefile`（SRCS 将自写 json_parser.c 替换为成熟 cJSON.c；新增 `test13` 目标）。

**与 V1.3 的关键对比**：

| 维度 | V1.3 (W3D3) | V1.4 (W3D4) |
|------|------------|------------|
| 配置来源 | 硬编码: port=8080, root=www | JSON 配置文件 server.json |
| 端口修改 | 需改代码 + 编译 | 改 config JSON 重启即生效 |
| 路由机制 | if/else 硬编码: `/search`、`/echo` | JSON routes 数组 → 运行时路由表 |
| Handler 绑定 | 函数名写死在 if 分支中 | name → fn 注册表查找 |

| 405 Allow 头 | 硬编码 "GET, POST" | 从路由表动态生成 |

| 日志路径 | 固定 logs/system.log + logs/access.log | JSON 可配，access 自动派生 |
| 向后兼容 | - | V1.3 serve-http 完全保持，V1.2/V1.1/V0.x 全部兼容 |

## 二、遇到的一个问题、原因和解决方案

**问题**：`send_response()` 依赖的静态 handler 表中 `WWW_ROOT` 宏被硬编码为 `"www"`，导致 V1.4 从 JSON 读取的 `document_root: "./www"` 能启动服务器，但静态文件实际仍从硬编码路径读取。更严重的是 `realpath()` 安全校验也基于硬编码的 `WWW_ROOT` 计算绝对路径前缀 —— 如果 config 中 `document_root` 指向其他目录（如 `./public`），`normalize_path()` 仍拼接 `www/` 前缀，而 `realpath("www/...", ...)` 与 `realpath(document_root)` 用不同根路径做跨域检查，导致路径穿越防护失效。

**原因**：`static_handler.c` 中共有 4 处直接使用 `WWW_ROOT` 宏：(1) `normalize_path()` 中 `memcpy(file_path, WWW_ROOT, root_len)` 拼接根路径前缀；(2) `serve_static_file()` 中 `realpath(WWW_ROOT, www_root_abs)` 获取绝对根路径用于越界检查；(3) `normalize_path()` 中空路径回退 `/index.html` 时比较 `file_path + strlen(WWW_ROOT)`；(4) `realpath()` 越界检查 `strncmp(safe_path, www_root_abs, root_len)` 基于绝对根路径前缀匹配。

**解决方案**：以模块级变量 `static char g_document_root[MAX_PATH_LEN] = "www"` 替代 `#define WWW_ROOT "www"` 宏，新增 `set_document_root(const char *root)` 和 `get_document_root()` API。V1.4 的 `http_server_run()` 在入口处调用 `set_document_root(config->document_root)`，所有原本引用 `WWW_ROOT` 的位置改为引用 `g_document_root`。对于 V1.3 `serve-http` 模式，默认值 `"www"` 保持不变，确保向后兼容。这个改动同时消除了"配置值和代码宏不一致"的隐患 —— 配置文件、变量副本、路径拼接、安全校验现在全部共享同一个来源。

## 三、还存在什么不足，下一步如何改进

1. **日志等级未实际过滤**：虽然 `server.json` 支持 `logging.level` 字段且校验了合法值（DEBUG/INFO/WARN/ERROR），但 `log.c` 模块的四个输出函数（`log_debug` / `log_info` / `log_warning` / `log_error`）并未根据配置等级过滤 —— 所有级别的日志都会被写入。改进：新增 `log_set_min_level(const char *level)` API，在 `sys_log_write()` 中按枚举值比较（DEBUG=0 < INFO=1 < WARN=2 < ERROR=3），低于阈值的消息静默丢弃。

2. **cJSON 的 `cJSON_Parse()` 默认大小写不敏感匹配对象键**：cJSON 的 `cJSON_GetObjectItem()` 默认使用大小写不敏感的键比较，意味着 JSON 中 `"Port"` 和 `"port"` 被认为是同一个字段。虽然对配置解析影响有限（JSON 规范推荐键保持一致），但不同于 JSON Schema 严格模式的行为。改进：对关键字段校验使用 `cJSON_GetObjectItemCaseSensitive()` 替代 `cJSON_GetObjectItem()`，确保 `"Port": 8080` 和 `"port": 9090` 不会被混淆。

3. **handler 注册表分散在两处**：`config.c` 中的 `HANDLER_REGISTRY` 数组用于配置校验时的名称查找，`http_server.c` 中 handler 包装函数负责实际请求处理。添加新 handler 需要同时修改两个文件（config.c 注册名称、http_server.c 实现函数），容易遗漏。改进：将注册表移至单独的 `handler_registry.c` 文件，数组每项同时包含 `{name, fn}` 对，config.c 和 http_server.c 共享同一份注册表，新增 handler 只需修改一个位置。

4. **无配置热重载**：修改 `server.json` 后必须重启服务器才能生效。对于生产环境，重启会导致正在处理的连接中断。改进：监听 SIGHUP 信号，收到信号后重新加载配置文件，对比新旧路由表，新增/删除路由动态更新，`document_root` 和日志路径可热切换，`host/port` 需重启（bind 已固定）。

5. **静态文件 MIME 表仍是线性扫描**：15 个条目的 MIME 映射表每次请求都做 `strcmp()` 线性扫描。虽然条目少影响小，但如果扩展到数百个扩展名性能会退化。改进：启动时按扩展名排序，使用二分查找（当前 15 条目扫描 8 次 vs 最多 15 次），或使用编译期完美哈希。

6. **访问日志路径自动派生不够灵活**：当前 V1.4 在 `main.c` 中通过替换 `.log` 扩展名为 `_access.log` 来派生访问日志路径，对不符合此约定的文件名（如 `server-logs.txt`）会生成 `server-logs_access.log.txt`。改进：在 `server.json` 的 `logging` 段增加可选的 `access_file` 字段，显式指定访问日志路径；未指定时保持派生行为或默认写入同一文件。
