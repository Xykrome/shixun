# Webserver V1.2 训练报告

## 一、今天完成了什么

今天完成了 W3D2 的实训内容：**构建 HTTP 静态文件服务器 V1.2 —— 将请求 URL 安全映射到 `www/` 目录，根据扩展名返回正确的 MIME 类型，分块发送文件内容**。

W3D1 解决了"服务器怎样收到并解析 HTTP 请求"，W3D2 解决的是"服务器怎样把请求安全地转换为正确的静态文件响应"。核心新增模块 `static_handler.h/c`（617 行）负责路径规范化、安全校验、MIME 映射和分块发送；`http_server.c`（593 行）重构掉硬编码的 `generate_response()`，所有 GET 请求统一交由 `serve_static_file()` 处理，POST 到非 `/echo` 路径统一返回 405。

**新增文件**：`include/static_handler.h`、`src/static_handler.c`（静态文件处理器），`www/css/style.css`、`www/js/app.js`、`www/img/logo.png`、`www/img/logo.jpeg`、`www/favicon.ico`（网站资源），`tests/test_day12.sh`（58 项自动化测试）。

**修改文件**：`src/http_server.c`（路由重构——V1.1 的 `generate_response()` 删除，GET 请求转由静态文件处理器接管，POST /echo 保留为动态路由兼容 V1.1；增加 `clock_gettime()` 请求耗时统计；版本号更新到 V1.2），`src/log.c`（`access_log` 增加 `mime_type` 参数，日志格式从 7 字段扩充到 8 字段），`include/log.h`（同步更新函数签名），`src/main.c`（更新命令行帮助和注释为 V1.2），`Makefile`（SRCS 增加 `static_handler.c`，新增 `test11` 目标）。

**与 V1.1 的关键对比**：

| 维度 | V1.1 (W3D1) | V1.2 (W3D2) |
|------|------------|------------|
| GET 路由 | 三个硬编码路由（GET /、GET /missing、POST /echo） | 全部 GET 请求交由静态文件处理器，从 `www/` 目录查找文件 |
| 响应生成 | `generate_response()` 在代码中拼接 HTML 字符串 | `serve_static_file()` 从文件系统读取真实文件并发送 |
| MIME 类型 | 硬编码 `text/html` 或 `text/plain` | 扩展名→MIME 查表映射（15 种），文本资源显式 `charset=utf-8` |
| 路径安全 | 无校验 | `normalize_path()` 拒绝 `..` 和空字节 + `realpath()` 验证真实路径在 document root 内 |
| 文件发送 | 无（不读取文件） | `read()` 8KB 固定缓冲区循环 + `send_all()` 可靠发送（处理部分 send） |
| 状态码 | 200/404/405 | 200/400/403/404/405/500 六种，按 errno 精准映射 |
| 访问日志 | 7 字段（时间/IP/方法/路径/版本/状态码/字节数） | 8 字段（+ MIME 类型） |
| 请求耗时 | 无 | `clock_gettime()` 测量，写入系统日志 |
| 网站资源 | 仅 `www/index.html`（占位） | 完整博客站点：HTML + CSS + JS + PNG + JPEG + ICO |

**静态文件处理器**（`static_handler.h/c`）实现了四个核心函数：

- `get_mime_type(path)`：提取文件扩展名（最后一个 `.` 之后），转为小写，查 15 项映射表返回 MIME 字符串。文本类型（html/css/js/json/txt）显式附加 `; charset=utf-8`，二进制类型（png/jpeg/gif/ico/zip/pdf）不加 charset，未知扩展名返回 `application/octet-stream`。

- `send_all(fd, data, len)`：循环调用 `send()` 维护已发送偏移量。`errno == EINTR` 时重试（被信号中断），其他错误返回 -1，`send() == 0`（对端关闭）也返回 -1。确保所有字节被发送，解决"一次 send() 不保证发送全部数据"的问题。

- `normalize_path(url_path, file_path, size)`：五步安全映射：(1) 复制 `www/` 根目录前缀；(2) 扫描原始路径，拒绝空字节和包含 `..` 的路径段（目录穿越检测）；(3) `/` 映射到 `/index.html`；(4) 逐字符复制路径，跳过重复的 `/`，丢弃 `?` 及之后的所有字符（查询参数去除）；(5) 空路径兜底映射到 `/index.html`。

- `serve_static_file(client_fd, url_path, &status_code, &mime_type, &body_bytes)`：完整 11 步静态文件响应流程。路径规范化 → `realpath(WWW_ROOT)` 获取根目录绝对路径 → `realpath(raw_path)` 解析文件真实路径 → 验证解析结果以根目录绝对路径开头（拒绝符号链接越界）→ `stat()` 获取文件元数据 → `S_ISREG()` 确保是普通文件（拒绝目录/设备/FIFO/socket）→ `get_mime_type()` 确定 MIME → 构造含 `Content-Type`、`Content-Length`、`Connection: close` 的 HTTP 响应头并用 `send_all()` 发送 → `open()` 文件 → 循环 `read()` 8KB 块 + `send_all()` 发送 → `close(file_fd)`。每一步失败都映射到正确的状态码（`..` 检测→403、`ENOENT/ENOTDIR`→404、`EACCES`→403、非普通文件→403、其他 I/O 错误→500），返回对应的 HTML 错误页面。

**响应头由文件元数据生成**：`Content-Type` 来自 MIME 映射表（非硬编码），`Content-Length` 来自 `st.st_size`（非 `strlen()`），保证二进制文件也能正确传输。分块发送内存占用仅 8KB，与文件大小解耦。

**安全机制**：三层防护——(1) `normalize_path()` 扫描阶段拒绝 `..` 和空字节；(2) 对存在的路径执行 `realpath()` 解析符号链接和相对路径；(3) 验证 `realpath()` 结果必须以 document root 绝对路径开头（如 `/home/wamdus/shixun/mini_webserver/www/`），不以根路径开头的返回 403。`curl --path-as-is http://127.0.0.1:8080/../etc/passwd` 返回 403 Forbidden。

**日志增强**：访问日志从 V1.1 的 7 字段格式扩充为 8 字段（增加 MIME 类型列）。新版格式示例：`2026-07-19 10:01:03 127.0.0.1 GET / HTTP/1.1 200 text/html; charset=utf-8 1695`。系统日志增加请求耗时记录（如 `0ms`），使用 `clock_gettime(CLOCK_MONOTONIC)` 精确测量。

对比验收标准全部检查点：编译与启动（`make clean && make` 无错误，`./mini_web_server serve-http 17` 成功监听 8080 端口）、GET / 返回首页（200 + Content-Type: text/html + 含 "Hello, HTTP!" + 引用 CSS）、GET /index.html 与 / 返回相同内容、CSS 资源（200 + Content-Type: text/css + Content-Length 与文件大小一致）、JS 资源（200 + Content-Type: text/javascript）、PNG 图片（200 + Content-Type: image/png + 无 charset）、JPEG 图片（200 + Content-Type: image/jpeg）、favicon.ico（200 OK）、404 处理（/missing → 404 + Content-Type 头存在 + 服务器不崩溃）、目录穿越防护（/../etc/passwd → 403 + 多层 .. 被拒绝）、405 方法检查（PUT → 405 + POST / → 405 + Allow: GET）、POST /echo V1.1 兼容（200 + 回显 "hello HTTP"）、多请求并发（3 个请求全部成功 + 服务器稳定运行）、请求数量控制（17 个请求后正常退出）、epoll API（epoll_create1/epoll_ctl/epoll_wait/LT+EPOLLIN 正确使用）、连接清理（EPOLL_CTL_DEL + close）、技术限制（未使用 select/多线程/多进程/线程池）、静态文件模块（5 个核心函数 + realpath + S_ISREG 均存在）、日志系统（双文件存在 + 访问日志含 MIME 类型 + 系统日志含级别和耗时）、三类资源均有访问记录（GET / 200 text/html、GET /css/style.css 200 text/css、GET /missing 404），全部通过，自动测试脚本 **58/58 项通过**，V1.1 回归测试 **38/38 项通过**。

## 二、遇到的一个问题、原因和解决方案

**问题**：`curl --path-as-is http://127.0.0.1:8080/../etc/passwd` 返回 400 Bad Request 而非验收标准要求的 403 Forbidden。

**原因**：在 `serve_static_file()` 的第一步——`normalize_path()` 路径规范化中，包含 `..` 的路径（目录穿越攻击）和包含空字节的路径都被统一返回 -1，调用方将 -1 统一按"路径格式非法"处理，返回 400 Bad Request。然而 HTTP 语义中，400 表示请求格式错误（客户端可重试），403 表示服务器理解请求但拒绝执行（权限/安全原因）。目录穿越攻击属于"路径越界"——服务器明确拒绝公开文档根目录之外的文件，应返回 403 Forbidden。400 不仅语义不准确，而且向攻击者透露了较少信息（应该明确告知"这个操作是不被允许的"）。

**解决方案**：在 `serve_static_file()` 的路径规范化失败分支中，增加对原始 URL 路径的判断——如果原始路径中包含 `..`（`strstr(url_path, "..") != NULL`），则将状态码设为 403 Forbidden，其余格式错误仍保留为 400 Bad Request。同时将原本只输出 400 错误页面的代码重构为根据状态码分支构造对应的响应行（`403 Forbidden` 或 `400 Bad Request`）和错误页面。修复后 `curl --path-as-is http://127.0.0.1:8080/../etc/passwd` 正确返回 `HTTP/1.1 403 Forbidden`，多层穿越 `../../etc/passwd` 同样被拒绝。

## 三、还存在什么不足，下一步如何改进

1. **不支持 HTTP 缓存机制（304 Not Modified）**：当前每次请求都重新发送完整文件，浏览器无法利用本地缓存。`www/` 中的 CSS、JS、图片等静态资源在多次访问时被重复传输，浪费带宽。改进：响应头增加 `Last-Modified`（文件 `st_mtime`）和 `ETag`（基于文件大小+修改时间的哈希），收到 `If-Modified-Since` 或 `If-None-Match` 请求头时比较时间戳/标签，未变化则返回 304 Not Modified 无响应体，节省带宽。

2. **不支持 HTTP 范围请求（206 Partial Content）**：当前无法支持断点续传和大文件分段下载（如视频拖动进度条）。浏览器请求视频文件时发送 `Range: bytes=0-1023`，服务器应返回 206 和 `Content-Range` 头，只发送指定范围的数据。改进：解析 `Range` 请求头，使用 `lseek()` 定位文件偏移，只发送请求的字节范围，响应行改为 `HTTP/1.1 206 Partial Content`。

3. **MIME 映射表为线性扫描（O(n)），且不支持大小写混合**：当前 15 项映射表使用 `strcmp` 逐个比较，每次请求 O(n)。扩展名 `.JPG`、`.Html` 等大小写变体依赖调用方先转小写。改进：使用哈希表或完美哈希将查找复杂度降为 O(1)；在 `get_mime_type()` 内部统一转小写后查表，彻底屏蔽大小写问题。

4. **文件发送期间阻塞整个事件循环**：当前 `serve_static_file()` 在发送文件内容时，`read()` + `send_all()` 循环在客户端 fd 上阻塞。大文件（如视频）或慢速客户端会长时间阻塞 `epoll_wait` 事件循环，导致其他客户端请求得不到处理。改进：将文件 fd 设为非阻塞，记录每个客户端的文件读取偏移和发送缓冲区，在 `epoll_wait` 中同时监听 `EPOLLOUT` 事件，每次可写时发送一个块，未发完则继续等待下次 `EPOLLOUT`，与 `EPOLLIN` 交替推进多客户端并发。

5. **不支持 HTTP Keep-Alive（持久连接）**：当前每个连接处理一个请求后立即关闭（`Connection: close`），浏览器加载首页需为 HTML、CSS、JS、PNG、ICO 分别建立 5 次 TCP 连接。改进：响应头改为 `Connection: keep-alive`，处理完一个请求后不关闭 client_fd，重置接收缓冲区继续在 epoll 中监听同一 fd，配合超时机制（如 60 秒无活动自动关闭）。

6. **访问日志缺少客户端端口和请求耗时**：当前访问日志记录客户端 IP 但不含端口号，多客户端同时访问时无法区分同一 IP 的不同连接；虽然系统日志记录了耗时，但访问日志中没有。改进：访问日志的客户端标识改为 `IP:PORT` 格式，并在行尾增加耗时字段（毫秒），方便性能分析和问题排查。
