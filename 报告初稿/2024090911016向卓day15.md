# Webserver V1.5 训练报告（W3D5 · Day15 · 含选做 Session + Bearer Token）

## 一、今天完成了什么

今天完成了 W3D5 的实训内容：**在 V1.4 配置驱动架构之上，新增三层认证中间件——HTTP Basic 认证（必做）、Session 会话认证（选做1）、Bearer Token 认证（选做2），实现"同一台服务器，不同路由采用不同安全策略"**。

W3D4 解决了"配置和路由从哪里来"，W3D5 解决的是"路由匹配之后、handler 执行之前，如何验证请求方身份"——认证发生在路由匹配之后、handler 之前，不修改 epoll 事件循环和 HTTP 请求解析。三种认证方案按路由独立配置：`auth: "basic"`、`auth: "session"`、`auth: "bearer"`，空值表示公开路由。

### 1.1 必做：HTTP Basic 认证

**新增文件**：`include/auth.h`（声明 `validate_basic_auth()` 和 `base64_decode()`）、`src/auth.c`（Base64 解码器 + Authorization 头解析 + 凭据比对 + 401/400 响应发送）、`www/secured.html`（受保护页面）。

**修改文件**：`include/config.h`（`route_config_t` 已有 `auth` 字段，`server_config_t` 新增 `basic_username` / `basic_password`）、`src/config.c`（`load_json_config()` 新增 `auth.basic` 段解析，`is_valid_auth_scheme()` 支持 basic/session/bearer，HANDLER_REGISTRY 新增 `secured_get` 等 6 个 handler）、`src/http_server.c`（`route_entry_t` 新增 `auth` 字段，dispatch 逻辑中新增认证中间件：`find_route()` → 检查 `auth` 字段 → `validate_basic_auth()` / `validate_session_auth()` / `validate_bearer_auth()` → handler；新增 `secured_get_handler` 等 7 个 handler 实现）、`config/server.json`（新增 `auth.basic` 凭据段 + `/secured` 路由标记 `auth:basic`）、`Makefile`（SRCS 新增 `src/auth.c`，新增 `test14` 目标）、`tests/test_day14.sh`（适配新路由数量）。

**核心实现**：
- `validate_basic_auth()` 流程：查找 Authorization 头（重复→400）→ 校验 Scheme 为 Basic → Base64 解码 → 按首个冒号分割 username:password → 与 config 凭据比对 → 失败统一返回 401 + `WWW-Authenticate: Basic realm="mini_webserver"`
- `base64_decode()` 自包含实现：跳过空白字符 → 4 字符 6-bit 查表 → 组合为 3 字节 → 处理 `=` 填充

### 1.2 选做1：Session 会话认证

**新增文件**：`include/session.h`（Session 存储结构 + Cookie 解析/构建 + CSRF Token 生成）、`src/session.c`（`/dev/urandom` 随机 SessionID 生成 → 32 字节 → 64 位十六进制 → 服务端内存查表 → `HttpOnly; SameSite=Strict` Cookie → 30 分钟超时懒清理 → 登录后刷新 SessionID 防会话固定）、`www/login.html`（Session 登录表单页面）。

**核心实现**：
- `session_create()`：从 `/dev/urandom` 读取 32 随机字节 → 生成 64 位十六进制 SessionID → 存储 `{session_id, username, csrf_token, created_at, last_access}` → 返回 `Set-Cookie` 头
- `session_lookup()`：遍历 `session_store` 数组 → 比对 session_id → 检查 `now - last_access > 1800s` → 过期自动标记 `active=0`
- `session_destroy()`：登出时设置 `active=0`，返回 `Set-Cookie: session_id=; Max-Age=0; Expires=Thu, 01 Jan 1970 00:00:00 GMT`
- CSRF 防护：生成独立随机 CSRF Token，POST/PUT/DELETE 请求需通过 `X-CSRF-Token` 请求头或 POST body 中的 `csrf_token` 字段提交，双重检测

**新增路由**：`GET /login`（登录页）、`POST /login`（Session 登录）、`POST /logout`（销毁 Session）、`GET /dashboard`（Session 保护的个人中心，`auth:session`）。

### 1.3 选做2：Bearer Token 认证

**新增文件**：`include/bearer.h`（Token 存储结构 + `parse_bearer_token()`）、`src/bearer.c`（Token 生成/查找/吊销/过期清理 + `Authorization: Bearer <token>` 头解析）。

**核心实现**：
- `token_create()`：生成随机 Token → 存储 `{token_id, username, issuer, audience, created_at, expires_at}` → 支持 `iss` (签发者)、`aud` (受众)、`exp` (过期时间) 三个 JWT 标准字段
- `token_lookup()`：查表校验 → 自动检查 `now > expires_at` 过期
- `parse_bearer_token()`：解析 `Authorization: Bearer <token>` 格式，校验 Scheme

**新增路由**：`POST /token`（获取令牌，返回 JSON：`access_token` / `token_type` / `expires_in` / `issuer` / `audience`）、`GET /api/me`（Bearer 保护的 API，`auth:bearer`）。

### 1.5 认证中间件架构

```
handle_request(req):
    path = normalize_path(req.target)
    route = find_route(req.method, path)
    if route exists:
        if route.auth == "basic"    → validate_basic_auth(req)    → fail → 401
        if route.auth == "session"  → validate_session_auth(req)  → fail → 401/403
        if route.auth == "bearer"   → validate_bearer_auth(req)   → fail → 401
        return route.handler(req)
    if method_not_allowed(path):
        return 405 + Allow header
    return try_static_file(path)
    return 404
```

## 二、遇到的一个问题、原因和解决方案

**问题**：Dashboard 页面点击 Logout 按钮后，Session 未被销毁，从 login 页面点击 Dashboard 链接仍能直接访问并显示用户信息。

**原因**：根因在 CSRF 校验环节。Dashboard 的 Logout 按钮是一个 HTML `<form method="POST" action="/logout">`，CSRF Token 放在隐藏表单域 `<input type="hidden" name="csrf_token" value="...">` 中。表单提交时，浏览器将 `csrf_token=xxx` 放在 POST body（`application/x-www-form-urlencoded`），但 `validate_session_auth()` 中间件的 CSRF 校验只在 `X-CSRF-Token` 请求头中查找 token。由于找不到 CSRF Token，校验返回失败（401），`session_logout_handler_impl()` 从未被执行，`session_destroy()` 未被调用，Session 保持有效。表面现象是"退出后还能访问 Dashboard"，实际是退出操作本身没有被正确处理。此外，`/logout` 路由配置为 `auth:session` 也存在设计问题——浏览器 POST 表单时如果因某些原因未携带 Cookie（如 SameSite 限制），中间件直接返回 401，handler 同样不执行。

**解决方案**：三处修改协同修复：(1) CSRF 校验改为双重检测——先查 `X-CSRF-Token` 请求头（适配 JS fetch 客户端），未找到则搜索 POST body 中的 `csrf_token=` 参数（适配 HTML 表单提交），使用 `strstr()` 定位 + `strchr('&')` 提取值 + `strcmp()` 比对；(2) 将 `/logout` 路由的 `auth` 从 `session` 改为空（公开路由），确保 handler 始终被执行——handler 内部自行从 Cookie 解析 session_id 并调用 `session_destroy()`，Cookie 存在就销毁，不存在也正常返回；(3) `build_clear_cookie_header()` 在原有 `Max-Age=0` 基础上增加 `Expires=Thu, 01 Jan 1970 00:00:00 GMT`，提高浏览器兼容性。

## 三、还存在什么不足，下一步如何改进

1. **Session 凭据硬编码**：Session 登录的 `session_login_handler_impl()` 中凭据比对直接硬编码 `strcmp(username, "student")` / `strcmp(password, "lab123")`，未使用 `config->basic_username/basic_password`。Basic 认证和 Session 登录各自维护了校验逻辑。改进：统一凭据校验入口，Session 登录复用 `server.json` 中 `auth.basic` 字段或新增 `auth.credentials` 独立段，消除代码中的硬编码字符串。

2. **Bearer Token 为不透明 Token，非标准 JWT**：当前实现为服务端查表的不透明 Token（类似 Session 的另一种形态），虽然 `token_entry_t` 预留了 `issuer`、`audience`、`expires_at` 等 JWT 标准字段，但未实现 JWT 的签名校验（HMAC-SHA256 / RS256）。改进：引入成熟 JWT 库（如 libjwt + OpenSSL），将 Token 从服务端查表模式升级为无状态 JWT 验证——客户端携带 `base64url(header).base64url(payload).signature`，服务端用密钥验证签名后直接信任 payload 中的 `sub/exp/iss/aud` 声明，无需查表，提升水平扩展能力。

3. **Session/Cookie 缺少 Secure 属性**：当前 Cookie 设置为 `HttpOnly; SameSite=Strict; Path=/`，但因本地测试使用 HTTP 而未设置 `Secure` 属性。改进：检测请求是否通过 HTTPS（检查 `X-Forwarded-Proto` 头或监听端口），对 HTTPS 请求自动添加 `Secure` 属性，防止 Cookie 在明文传输中被窃听。

4. **登录失败无限流/无锁定机制**：当前 Basic 认证和 Session 登录均未对失败尝试进行限流，攻击者可在短时间内发起大量暴力破解尝试。改进：基于 IP + 用户名维护失败计数器（固定窗口或滑动窗口），连续失败 5 次后锁定 15 分钟或要求延时重试，计数器存储在内存中（重启丢失，生产环境应使用 Redis）。

5. **CSRF Token 仅在中间件校验，未在页面渲染时注入**：当前 CSRF Token 由 Dashboard handler 在生成 HTML 时直接嵌入 `<input type="hidden">`，但其他 session 保护的 POST handler（如未来新增的 `/settings` POST）需要各自在响应中嵌入 CSRF Token。改进：将 CSRF Token 注入从 handler 层提升到中间件层——在所有 session 认证的响应中自动注入 `X-CSRF-Token` 响应头或统一处理 CSRF Token 的页面嵌入。

6. **三种认证方案缺少统一抽象**：`validate_basic_auth()`、`validate_session_auth()`、`validate_bearer_auth()` 三个中间件函数各自独立实现了错误响应发送（401 HTML / 401 JSON），代码重复较多。改进：定义统一的认证中间件接口 `typedef int (*AuthMiddleware)(int client_fd, const http_request_t *, ...)` 和认证失败响应构造器，将 auth scheme → middleware 的映射表化，新增认证方案只需注册一行映射。
